Project History & Evolution

The application now documented as "Music Library Manager" began life under a different name, survived a four-name identity crisis, absorbed a sweeping code-quality cleanup, and then a focused security hardening wave — before settling into a self-sustaining, multi-agent development pipeline.

2.3 Era 0 — Origin: "RB Editor Pro"

The application was originally named RB Editor Pro — "RB" for Rekordbox. From the very beginning it was a three-stack desktop application: a Tauri 2 shell wrapping a React frontend and a Python FastAPI backend. Even in this earliest recoverable state the project already had its defining features in place, as the HANDOVER document records:

  • Rekordbox XML library handling.
  • A non-destructive audio editor.
  • SoundCloud synchronisation built on an OAuth PKCE flow.
  • USB synchronisation to CDJ hardware.
  • FFT-based waveform rendering.

The repository lived locally at a Windows path ending in RB_Editor_Pro, and the project was tracked on GitHub. This was a working product — but it carried the scars of fast, exploratory development, which the next three eras systematically removed.

2.4 Era 1 — The Identity Crisis and the Rename

By early May 2026 the project had a genuine problem: it answered to four different names depending on where you looked.

LocationName it used
Local filesystem pathRB_Editor_Pro
Root package.jsonmusic-library-manager
The GitHub repositoryLibraryManagementSystem
Cargo.tomllibrary-management-system

This is more than cosmetic. A name appears in load-bearing places: the Tauri bundle identifier (wrong value breaks the installer), and the OS keyring service name used to store SoundCloud OAuth tokens (wrong value breaks login silently). To untangle it without breaking anything, the team produced docs/NAMING_MAP.md on 2026-05-05 — an exhaustive, line-numbered inventory of every name occurrence, classified by criticality.

▲
The critical occurrences

Nine keyring calls in app/main.py used the service name "rb_editor_pro" — and the map flagged that any existing user's stored tokens would become unreachable after the rename, calling for an optional migration helper. The Tauri bundle identifier com.rbeditor.pro was likewise marked as installer-breaking.

The refactor standardised on library-management-system (kebab-case for manifests, snake_case where a manifest required it) and was recorded as the work of Alpha v0.0.2. The naming map was deliberately retained in the repository afterwards — not as dead weight, but as an audit trail and as a recipe for any future rename. It even ends with a step-by-step procedure for the next time the project changes its name.

2.5 Era 2 — The v1.0.0-beta Release

On 2026-05-07 the project shipped its first public beta, v1.0.0-beta. The changelog frames it precisely: a "standalone DJ-library manager that competes with Rekordbox/Serato while staying open and local-first." The beta's defining achievement was that both headline workflows from Chapter 1 were now fully supported end to end.

What the beta could do

Workflow ASoundCloud OAuth, playlist and likes browser, per-playlist download with an anonymous fallback, full auto-analysis, auto-import, auto-playlist, USB sync.
Workflow BDrag-drop and folder-picker import; the folder name becomes the playlist; the identical analysis and ANLZ pipeline.
LibraryStandalone XML mode with full feature parity to Live mode; a wrapper routing every CRUD operation to the active mode.

The changelog is also honest about the beta's limits: USB sync depended on pyrekordbox, manual hot-cue editing was post-MVP, and creating a master.db from scratch without a Rekordbox install was not yet possible. The beta also documented an "Aggressive Download Mode" — a hidden setting, revealed by tapping a dot five times, intended only for tracks a user has a personal right to.

2.6 Era 3 — The Slopcode-Cleanup Mission

A beta that works is not the same as a beta that is maintainable. Around 2026-05-11 the team commissioned a sweeping, systematic quality remediation, briefed in docs/HANDOVER.md as the Slopcode-Cleanup. The handover document is itself a notable artefact: a complete mission briefing written for a fresh AI instance with no prior context, including hard constraints, definitions of done, and an escalation protocol.

The cleanup was structured as six sequential phases, to be completed and committed one at a time:

What the cleanup found and fixed

The handover's audit appendix is a frank inventory of the technical debt a fast beta accumulates. A representative sample:

LayerDebt the audit catalogued
Backend40+ bare except: clauses, three raw SQL f-strings, ten synchronous HTTP calls inside async functions, three subprocesses with no timeout, a 207-line exception handler, ~87 missing type hints across three modules.
RustAn unsafe impl Send + Sync, a detect_key() that returned a hardcoded fake value, an FFT band bug treating bin indices as Hz, ~50 println! calls, undocumented commands.
Frontend24 alert/confirm/prompt calls, 34 stray console.log statements, five raw fetch() calls, and seven "god components" — the largest, WaveformEditor.jsx, was 2,223 lines.
RepositoryThe four-name chaos, dead Docker files, 33+ debug scripts tracked in app/, a drifting requirements.txt, and no linters, formatters, or CI at all.

The cleanup's lasting legacy is visible all over the codebase today. The seven god-components were split into the focused sub-component directories documented in Chapters 15–17. The Python tooling (ruff, black, mypy), the frontend tooling (ESLint, Prettier), and the ci.yml workflow that gates every push all date from Phase 5. The structured logging, the typed exceptions, and the concurrency locks documented in Chapters 7–10 were all installed here. This era is the reason the rest of this handbook can describe a clean, layered, rule-bound codebase rather than a beta-grade one.

2.7 Era 4 — The Security Hardening Wave

Between 2026-05-16 and 2026-05-19 the project executed a focused security campaign. It began as a single audit of the FastAPI route surface and fanned out into seven distinct, individually-researched and individually-shipped security features. Each one has an archived research document recording the problem, the options considered, and the outcome.

ShippedFeatureWhat it closed
05-17API auth hardeningThe parent feature. Bearer-token auth on 84 of 85 mutation routes; the legacy SHUTDOWN_TOKEN deleted.
05-17Constant-time compareA shared safe_compare helper replacing timing-attack-prone == on secrets.
05-17Rate-limit designA custom 180-line token-bucket limiter on the three highest-risk routes.
05-18File-reveal sandboxA path-traversal hole where /api/file/reveal could open any location on disk.
05-18Error-handler leakStack traces with absolute paths persisted to the log file.
05-18Pydantic blob-writeAn unbounded stored-blob write primitive in the settings endpoint.
05-19CORS tighteningWildcard CORS and a latent CSRF surface; allow_credentials turned off.

This wave is significant enough — and the resulting architecture load-bearing enough — that all of Chapter 21 is devoted to the security model it produced. What matters for the history is the method: each fix was treated as a research topic with a written rationale, not a hurried patch. That method became permanent in the next era.

2.8 Era 5 — The Multi-Agent R&D Pipeline

On 2026-05-20 the project formalised the way it does development. The security wave had proven that "research the problem, write down the options, then implement" produced better results than diving straight into code. The team turned that practice into infrastructure: a multi-agent research-and-implementation pipeline.

Every feature now lives as a Markdown document that moves through a fixed sequence of states, its current state encoded in its folder and filename prefix. Five autonomous routines, running on a cron schedule, advance those documents — drafting, researching, planning, and even writing code — while the user keeps four deliberate sign-off gates.

flowchart LR I["idea_"] --> D["drafting_"] D --> GA{"GATE A"} GA --> E["exploring_"] E --> GB{"GATE B"} GB --> V["evaluated_"] V --> DP["draftplan_"] DP --> R["review_"] R --> GC{"GATE C"} GC --> AC["accepted_"] AC --> IP["inprogress_"] IP --> GD{"GATE D"} GD --> IM["implemented_"]

The git log of this era reflects the transition directly: commits adding the five versioned routine prompts, the pipeline status script, the gate slash-commands, and the documents that retarget the project's own operating rules to the new model. The pipeline is not a side project — it is now the primary way the roadmap in Chapter 25 gets built.

Chapter summary

The project evolved through six eras: an origin as "RB Editor Pro", a v0.0.2 rename that reconciled four conflicting names, the v1.0.0-beta release of 2026-05-07, a six-phase Slopcode-Cleanup that installed the tooling and discipline the codebase runs on today, a four-day security hardening wave that shipped seven features, and finally a multi-agent research pipeline that turned "research before code" into permanent infrastructure. The team's habit of keeping process artefacts in-tree is why this history can be told at all.
