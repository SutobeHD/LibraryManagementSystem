System Architecture

Music Library Manager is a three-tier desktop application. A Rust/Tauri shell hosts a React user interface and a native audio engine, and supervises a bundled Python web server that runs entirely on the user's own machine.

4.1 The Shape of the System

The application looks, to the user, like a single desktop program. Inside, it is four cooperating parts written in three languages. The architecture is best understood as a three-tier model wrapped in a desktop shell:

4.2 Why Three Languages

Using three languages is a deliberate choice, not an accident of history. Each layer is placed where its language is strongest, and the coding rules forbid crossing those boundaries without a reason.

ConcernLives inWhy
Real-time audio (playback callbacks)RustA garbage-collected language cannot guarantee the microsecond-level determinism an audio callback needs. Rust has no GC and zero-cost abstractions.
Offline DSP (BPM, key, beatgrid)PythonThe mature analysis ecosystem — librosa, madmom, essentia, scipy, numpy — is Python. Re-implementing it would be reckless.
HTTP, orchestration, databasePythonFastAPI plus the pyrekordbox library make Python the natural home for the library and the API.
The user interfaceReactA rich, interactive desktop UI is exactly what a component framework like React is built for.
◆
The boundary rule

The project's coding rules state it plainly: "No layer-crossing without reason. No reimplementing Python in JS or vice versa." Realtime audio is Rust; offline DSP is Python; UI is React. When a feature seems to want code in two places, that is a signal to rethink the design, not to duplicate logic.

4.3 The Sidecar Pattern

The single most distinctive architectural decision is that the Python backend is a sidecar — a child process that the Rust shell starts, supervises, and kills. The backend is a real web server, with real REST endpoints, but it serves exactly one client (the local frontend) and binds exclusively to the loopback interface 127.0.0.1.

This pattern delivers several things at once:

  • Language freedom. The UI gets React; the analysis engine gets Python's DSP libraries. Neither is compromised.
  • A clean contract. The frontend and backend talk over HTTP — a well-understood, debuggable, testable boundary.
  • Process isolation. If the Python process misbehaves, it can be restarted without touching the window or the audio engine.
  • Lifecycle ownership. The Rust shell starts the sidecar on launch and kills it on exit, so there are never orphan processes.

The backend is packaged for release as a standalone executable (rb-backend) built with PyInstaller, so the end user never installs Python. Chapter 22 covers that packaging in detail.

4.4 Two Communication Channels

The frontend is the hub. It talks to the other two active parts over two distinct channels, and which channel it uses depends on what it needs.

flowchart LR FE["React Frontend"] FE -->|"HTTP REST · axios
library, playlists, analysis,
USB, SoundCloud, settings"| BE["Python Backend"] FE -->|"Tauri IPC · invoke()
audio playback, waveform,
OAuth, file dialogs, devices"| RUST["Rust Native Core"] RUST -->|"events
export_progress, sc-login-progress"| FE

Channel 1 — HTTP REST to the backend

Anything involving the library, the database, analysis, USB export, or SoundCloud goes over HTTP to the FastAPI sidecar. The frontend uses a single shared axios client (Chapter 14) that attaches an authentication token to every request. There are 146 such routes, catalogued in Chapter 8.

Channel 2 — Tauri IPC to the Rust core

Anything that needs the operating system or low-latency audio goes over Tauri's IPC bridge. The frontend calls invoke('command_name', args) and receives a Result back; the Rust side can also push events the frontend listens for. Native file dialogs, audio device enumeration, waveform FFT, and the SoundCloud OAuth flow all travel this channel. Chapter 18 documents every command.

▲
Why not one channel?

It would be possible to route everything through HTTP. But native file pickers, OS audio devices, and the OAuth browser callback genuinely require host-process privileges the WebView does not have — and realtime audio cannot tolerate an HTTP hop. The two-channel split puts each request on the path with the right capabilities and the right latency.

4.5 The Boot Sequence

When the user double-clicks the application, a precise sequence unfolds. Understanding it explains both the splash screen and the authentication bootstrap.

sequenceDiagram participant OS as Operating System participant T as Tauri Shell (Rust) participant P as Python Sidecar participant W as WebView (React) OS->>T: launch executable T->>T: create splashscreen window T->>P: spawn sidecar process P->>P: generate SESSION_TOKEN P-->>T: stdout "LMS_TOKEN=..." T->>T: capture token, scrub from logs P->>P: Uvicorn binds 127.0.0.1:8000 T->>W: create main window, load frontend W->>T: invoke('get_session_token') T-->>W: the captured token W->>P: GET /api/system/health (with Bearer) P-->>W: 200 OK W->>T: invoke('close_splashscreen') T->>T: hide splash, show main window

Two details of this sequence are load-bearing and recur throughout the handbook:

  • The Python sidecar generates its own session token at boot and emits it as its very first line of standard output. The Rust shell captures that line, stores the token, and removes it from the log stream so it never lands in a log file. This is the foundation of the authentication model (Chapter 21).
  • The splash screen exists precisely because the sidecar takes a moment to start. The main window is not shown until the frontend has confirmed the backend is alive.

4.6 Development Mode vs Production Mode

The same architecture runs in two shapes. In development the pieces are loose and hot-reloading; in production they are bundled into one installed application.

AspectDevelopmentProduction
FrontendVite dev server on port 5173, hot module reload.Static frontend/dist bundled into the app.
Backendpython -m app.main — the interpreter runs the source.A PyInstaller-built rb-backend executable.
Token handoffVite middleware re-exposes the token file at GET /dev-token.The Rust shell captures the token from the sidecar's stdout.
Launchnpm run dev:full (browser) or npm run tauri dev (desktop).The installed desktop binary.

A subtle but useful property: even a directly-launched debug build behaves like a full development run — if ports 8000 and 5173 are free, the debug build auto-spawns both the Python backend and Vite. The application is also runnable purely in a browser via npm run dev:full, which is how much of the automated UI testing is done (Chapter 23).

4.8 Concurrency & State

Three facts about how the system handles concurrent work shape the backend chapters that follow:

MechanismPurpose
_db_write_lock (a re-entrant lock)Serialises every writer of the Rekordbox master.db so concurrent FastAPI request threads cannot corrupt it.
ProcessPoolExecutor for analysisHeavy DSP runs in worker processes, keeping the API responsive.
Subprocess isolation for ANLZ parsingA third-party library with a known crashing bug is quarantined in a single-worker subprocess so a crash kills only the worker.

The first and third of these are direct consequences of integrating with a real, imperfect third-party ecosystem. They are documented in depth in Chapter 10.

4.9 Performance Characteristics

The architecture's performance budget, as the project measures it:

OperationMechanismLatency
Library load — 10k tracksIn-memory XML cache~200 ms first load, <5 ms cached
Waveform renderRust FFT~50 ms for a 5-minute track
BPM analysisPython process pool2–8 s per track, in the background
Audio playback startMemory-mapped file + CPAL<50 ms
USB sync — 1000 tracksIncremental diffMinutes — I/O bound

The pattern is consistent: anything a user waits on synchronously is fast (cache hits, native FFT, memory-mapped playback), and anything genuinely slow (analysis, USB I/O) is pushed to a background worker so the UI never blocks.

Chapter summary

The system is a three-tier desktop application: a Rust/Tauri shell hosts a React UI and a native audio engine, and supervises a bundled Python FastAPI sidecar bound to loopback. The frontend talks to the backend over HTTP and to the Rust core over Tauri IPC. Three languages are used deliberately — Rust for realtime audio, Python for offline DSP and orchestration, React for the UI — and the boundaries are enforced by rule. The sidecar pattern, the boot sequence, and the concurrency locks introduced here are unpacked in Part III.
