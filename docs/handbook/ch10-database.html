Chapter 10 · The Backend

The Database Layer

The library can live in an XML file or in a Rekordbox SQLite database, and a uniform abstraction hides which. This chapter also covers the concurrency lock that keeps writes safe and the subprocess quarantine that keeps a buggy library from crashing the backend.

10.1 The Storage Picture

"The database" is really several stores. The user's library is one of two forms; alongside it sit a handful of small SQLite databases for deduplication, caching, and supplementary data.

10.2 The Two Library Modes

The application can manage the collection in either of two modes. The choice is the user's, made at the four-card mode picker on startup.

Standalone XML mode — RekordboxXMLDB

app/database.py defines RekordboxXMLDB, the default mode. It parses a Rekordbox-compatible XML document into an in-memory dictionary cache. Reads are then served from RAM — a 10,000-track library loads in roughly 200 ms and every subsequent read is a sub-five-millisecond cache hit. Writes mutate the cache; an explicit save_xml() persists the cache back to the file.

Live mode — LiveRekordboxDB

app/live_database.py defines LiveRekordboxDB, which reads and writes a real Rekordbox master.db SQLite database through the pyrekordbox library. It is thread-safe and takes an automatic backup before a write. Crucially, it opens a per-thread database handle — each FastAPI request thread gets its own rbox.MasterDb object — which makes the write lock of the next section essential rather than optional.

▲
Live mode requires Rekordbox to be closed

SQLite does not tolerate two processes writing one database file. LiveRekordboxDB writes refuse to proceed if Rekordbox itself has the file open — the backend checks via SystemGuard and returns the REKORDBOX_RUNNING error code.

10.3 The Uniform Abstraction

No feature in the codebase should contain an if xml_mode … else live_mode … branch. Two layers ensure that:

  • A wrapper routes every create/read/update/delete operation to whichever mode is active, so the route handlers call one API regardless of mode.
  • app/library_source.py defines LibrarySource, a uniform read abstraction over both modes used specifically by the USB sync engine — it can read tracks, playlists, and cues without caring where they come from.

This is why the changelog can claim "full feature parity" between Standalone XML and Live mode: parity is structural, not the result of duplicating each feature twice.

10.4 The Concurrency Model

FastAPI serves requests on a thread pool. With Live mode opening a handle per thread, two threads can issue two writes to one SQLite file at the same instant — and pyrekordbox 0.1.7 crashes the entire Python process when that happens. The backend's defence is a single re-entrant lock.

flowchart TB T1["Request thread A"] --> L{"_db_write_lock
(threading.RLock)"} T2["Request thread B"] --> L T3["Request thread C"] --> L L -->|"one at a time"| W["master.db write"]

The lock is applied automatically. At import time, a loop in database.py walks the mutating methods of the RekordboxDB class and wraps each one in a @_serialised decorator that acquires _db_write_lock before the method body runs. Because the lock is re-entrant, a wrapped method calling another wrapped method does not deadlock. Routes that need several mutations to be atomic together use an exposed db_lock() context manager rather than relying on method-by-method acquisition.

◈
A known coverage gap — on the roadmap

An empirical audit (the db-write-lock-retrofit research doc, Chapter 25) found the auto-wrap is a hand-maintained method-name list that can silently drift, and that the MyTags routes write through LiveRekordboxDB directly, bypassing the wrapped facade — three unprotected write paths. The proposed fix replaces the name list with a prefix-matched class decorator applied to both classes, plus a CI test that fails if any mutator is left unwrapped.

10.5 The ANLZ Subprocess Quarantine

The second great reliability problem with pyrekordbox is more severe than concurrency. The library has a known bug — an unwrap() on a None value, deep inside its Rust code — that aborts the whole Python process when it meets a malformed analysis row. On Windows the backend simply vanishes with exit code 0xC0000409. A single bad track in a user's library could kill the application.

The answer is app/anlz_safe.py and its SafeAnlzParser — a fortress of defences around every pyrekordbox ANLZ call.

to find the bad id"] BISECT --> BLACK["Blacklist offending id

The layered defences:

DefenceWhat it does
Header validation_validate_anlz_header() checks for the PMAI magic bytes and a minimum 28-byte size, rejecting truncated or non-ANLZ files before pyrekordbox ever sees them.
Subprocess isolationA ProcessPoolExecutor with one worker runs every pyrekordbox call. A panic kills only that worker; the parent respawns it.
Batch & bisectBeatgrids load in chunks of 500 tracks. When a worker crashes, the chunk is bisected — roughly log₂N respawns — to identify the single offending track ID.
Blacklist cacheAny track ID that panics, times out, or fails the header check is added to _bad_ids and skipped for the rest of the session.
Chunk timeoutA 60-second per-chunk timeout kills a hung worker — a hang is dropped without bisecting, since bisecting a hang would only hang again.
Panic budgetA hard ceiling of 32 subprocess restarts per run protects against a pathological database where every other row panics.

The result: the UI sees its tracks immediately, beatgrids fill in within seconds on a background daemon thread, and no malformed analysis file anywhere in a user's library can take the backend down. This is the "defensive isolation" principle of Chapter 1 in its purest form.

◆
Background, not blocking

Beatgrid loading is dispatched to a daemon thread after the library reports itself loaded. A slow or crashing pyrekordbox call therefore can never block library startup — the tracks appear, and the beatgrids catch up.

10.6 The Download Registry

app/download_registry.py is a dedicated SQLite database — at download_registry.db — that backs the SoundCloud feature with deduplication and a history log.

erDiagram DOWNLOAD { int sc_track_id PK string sha256 string status string device_id string file_path datetime downloaded_at } ANALYSIS_HISTORY { int id PK int sc_track_id FK float bpm string key string status datetime analyzed_at } DOWNLOAD ||--o{ ANALYSIS_HISTORY : "logged for"

The registry deduplicates on two axes: an O(1) lookup by SoundCloud track ID, and a content-hash lookup by SHA-256 for files that arrived by a different route. It also records a stable per-machine device ID — stored in a .rb_device_id file, not hardware-tied — so a multi-device DJ can see which machine downloaded what. Chapter 20 covers how the SoundCloud downloader uses it.

10.7 The Analysis Cache

app/analysis_cache.py is a persistent cache that avoids re-running expensive analysis on a file that has not changed. Its validity check is a triple — file modification time, file size, and the analyzer version — and only if all three match is a cached result reused.

Index
index.json — maps file path to {mtime, size, cache id, version}
Per-file result
<cache_id>.json.gz — the gzipped analysis result
Invalidation
Any change to the analyzer's output schema bumps a version constant, invalidating every stale entry

Content hashing is available as an opt-in (USE_CONTENT_HASH) for "paranoid setups", but it is off by default because hashing a large file is slow — the mtime/size/version triple is fast and sufficient for the common case.

Chapter summary

The library lives in one of two modes — an in-memory-cached XML file or a Rekordbox master.db — behind a uniform abstraction so no feature branches on mode. A re-entrant _db_write_lock serialises every writer, because pyrekordbox crashes on concurrent writes; a single-worker subprocess with header validation, bisecting, blacklisting, and a panic budget quarantines the library's process-aborting parsing bug. Auxiliary SQLite stores handle download deduplication and analysis caching.
