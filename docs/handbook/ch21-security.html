Security Architecture

A loopback-only desktop application still has a real attack surface. This chapter documents the defence-in-depth model — Bearer authentication, a filesystem sandbox, rate limiting, log redaction, and supply-chain hardening — built in a focused four-day campaign.

21.1 Why a Local App Needs Security

It is tempting to think a desktop application bound to 127.0.0.1 needs no security at all. The project rejects that thinking. The backend is a real HTTP server, and a real HTTP server on a shared machine has real threats: same-machine malware can POST to it, a malicious local web page can try to pivot through it, and the moment anyone exposes the port to a LAN — for a mobile companion, a headless install, or by accident — the surface widens dramatically.

So the project treats security as a first-class concern. Chapter 2 described the four-day hardening wave of May 2026; this chapter documents the architecture that wave produced.

21.4 Bearer-Token Authentication

The centrepiece of the model is Bearer-token authentication on every mutation route — 84 of the 85 POST/PUT/PATCH/DELETE endpoints. The token is born, handed off, and verified through a precise mechanism.

The token's life

flowchart TB BOOT["Sidecar boot — app/auth.py"] BOOT --> GEN["SESSION_TOKEN = secrets.token_urlsafe(32)"] GEN --> EMIT["Emit 'LMS_TOKEN=...' on stdout — once"] GEN --> FILE["Persist to user-data dir
.session-token, chmod 0600 on POSIX"] EMIT --> CAP1["Tauri: Rust supervisor captures + scrubs"] FILE --> CAP2["Browser dev: Vite middleware → GET /dev-token"] CAP1 --> FE["Frontend obtains the token"] CAP2 --> FE FE --> HDR["Authorization: Bearer on every request"]

At boot, app/auth.py generates a 32-byte URL-safe random token. It emits the token once on standard output as LMS_TOKEN=<value> and persists it to a file in the cross-platform user-data directory. In a Tauri build, the Rust supervisor captures the stdout banner and scrubs it from the logs (Chapter 18); in browser-development mode, the Vite dev server reads the file and re-exposes it at GET /dev-token. Either way the frontend obtains the token and attaches it to every request as an Authorization: Bearer header.

The verification

On the server, require_session() is a FastAPI dependency declared on every mutation route. It parses the Authorization header, validates the scheme, strips and length-checks the token, and compares it against SESSION_TOKEN. Any failure — missing header, wrong scheme, length mismatch, value mismatch — raises a 401.

◈
Constant-time comparison

The comparison uses safe_compare() from app/security_compare.py — a hardened wrapper around secrets.compare_digest. A naive == on a secret leaks information through how long it takes to fail: it returns faster when the first character is wrong. A constant-time compare always takes the same time. safe_compare additionally returns False — never raising — for five fragility cases the audit identified: non-string inputs, non-ASCII strings on either side, mixed string/bytes, and length mismatch.

The one intentional exception is POST /api/system/heartbeat — a body-less liveness probe that returns only {"status":"alive"} and leaks no token. Read-only GET routes are loopback-gated only in this phase; making them require authentication too is explicitly deferred to a future phase.

▲
Never log the token

A permanent, codified rule: the token must never be logged — not at INFO, not at DEBUG, not redacted. app/auth.py enforces this and the coding rules require all future code to honour it. Any logging middleware added later must scrub Authorization and Cookie headers before emitting.

21.5 Rate Limiting

The three highest-risk routes — shutdown, restart, and soundcloud/auth-token — carry a second gate: a rate limiter in app/rate_limit.py, a custom 180-line token-bucket implementation chosen over adding a third-party dependency.

ElementBehaviour
TokenBucketA single-key bucket with monotonic-clock refill — take() returns whether the request is allowed and a retry-after.
BucketStoreA process-wide, lock-guarded, TTL'd dictionary of buckets, lazily purged.
make_key()Derives a bucket key — per-IP, per-bearer (hashed, never the raw token), or both. Loopback IPs short-circuit to a whitelist sentinel.
@rate_limitThe decorator — on a miss it raises 429 with a Retry-After header.

The decorator order matters: require_session runs before the rate-limit body, so a request with no Bearer token returns 401 without consuming a token from the bucket — verified by a dedicated test.

21.6 CORS

Cross-Origin Resource Sharing is locked down. The allowed origins are an explicit list — localhost, 127.0.0.1, and tauri://localhost — never a wildcard. The allowed methods and headers are explicit lists too. And as of the Phase B tightening, allow_credentials is False and the dead SoundCloud sentinel cookie was deleted: the Bearer-in-header is the only authenticated transport, and there is no ambient cookie for a CSRF attack to ride. A coding rule forbids any future PR from introducing cookie-based auth.

21.7 The Filesystem Sandbox

Many routes accept a file path from the client. validate_audio_path() is the gate: it resolves the path and checks it against ALLOWED_AUDIO_ROOTS with Path.is_relative_to() — not a string prefix check, which .. and symlinks defeat. A path outside the sandbox is refused.

◆
The file-reveal fix

The May-18 hardening found a real hole: POST /api/file/reveal passed a user-controlled path straight to the OS file manager with only an existence check — an authenticated caller could make the file manager navigate to anywhere on disk, including the token file or SSH keys. The fix routed the path through validate_audio_path() before the platform branch, and added a seven-case security test.

21.8 Input-Size Caps

The settings endpoint writes to a persistent store, which makes it a potential denial-of-service primitive — a buggy or compromised client could bloat settings.json until the disk fills and the UI hangs. The fix gave the SetReq Pydantic model strict caps: 8 KB per value, 64 dictionary keys, a 64-character key limit, 256 list items, and 256 KB total. A cap violation returns 422 and logs a warning that records the sizes but never the offending value.

21.9 Log Redaction

The May-18 work also closed a quieter leak: the global exception handler logged stack traces containing absolute filesystem paths to app.log — paths that then travelled into crash reports, antivirus ingestion, shared screenshots, and support bundles. app/logging_utils.py introduced a RedactingFormatter that strips absolute paths — under the export, music, and temp directories — from every log line and traceback at write time, across all handlers.

21.10 Supply-Chain Hardening — "Schicht A"

Beyond runtime defences, the project hardens its dependencies. The "Schicht A" policy treats the supply chain as an attack surface.

ThreatDefence
Postinstall malwareignore-scripts=true in .npmrc — disables npm install scripts, blocking the most common supply-chain vector.
Caret-version silent driftsave-exact=true plus exact pinning everywhere — no version moves without a review.
Lockfile manipulationlockfile-lint — every package must come from the npm registry over HTTPS.

Slopsquatting (AI-suggested non-existent packages)A manual package-verification workflow before any dependency is added.
Known CVEsnpm audit, pip-audit, cargo audit, and weekly Dependabot checks.

The policy is honest about what it does not cover: an XZ-style backdoor in an upstream source, a targeted attack on this specific project, and bugs in the project's own code (a separate concern, handled by review and the linters). One accepted risk is documented in detail — a ReDoS in picomatch, a build-time-only transitive dependency with zero production impact, with Dependabot watching for the upstream patch.

21.12 Accepted Risks and Future Work

The model is explicit about its current limits. Among the accepted Phase-1 risks: on Windows, the POSIX chmod 0600 on the token file is a no-op, so any local user with read access to the user-data directory can lift the token; the token sits in process memory for the supervisor's lifetime; and the token rotates only on a sidecar restart, with no per-device revoke.

The roadmap (Chapter 25) addresses the most significant of these: a Phase-2 paired-device token system — per-device 256-bit tokens, a QR-pairing flow, a sidecar-local auth.db, and a revoke UI — which is also the hard prerequisite for the mobile companion. The coding rules already codify the permanent constraints any future code must honour: scrub auth headers in any logging middleware, authenticate WebSocket routes inside the handler, and never log the token.

Chapter summary

Security is defence in depth: a loopback bind, locked-down CORS, a token-bucket rate limiter, Bearer-token authentication with constant-time comparison on 84 mutation routes, Pydantic input-size caps, a Path.is_relative_to filesystem sandbox, and log redaction. The supply chain is hardened under the "Schicht A" policy with exact pinning, disabled install scripts, and signature verification. Seven features were shipped in a four-day campaign, each researched and tested; the model's documented limits are addressed by the Phase-2 paired-token work on the roadmap.
