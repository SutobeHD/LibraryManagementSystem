Chapter 12 · The Backend

The ANLZ Binary Format

For a CDJ to show a track's waveform, beatgrid, and cues, it reads a set of per-track binary analysis files. This chapter documents the ANLZ format and the writer that produces it byte-for-byte.

12.1 What ANLZ Files Are

When Rekordbox analyses a track, it does not store the waveform and beatgrid inside the audio file or only in its database. It writes a small set of companion ANLZ files — "analysis" files — one set per track. A CDJ-3000 reads these directly off the USB stick to draw the coloured waveform, place the beat markers, and show the cue flags.

To export a stick a CDJ can actually use, the application must write these files itself, in exactly the format Rekordbox produces. That is the job of app/anlz_writer.py.

FileCarries
ANLZ0000.DATThe core file — path, beatgrid, the basic mono waveforms.
ANLZ0000.EXTThe extended file — high-resolution and colour waveforms.
ANLZ0000.2EXThe second extended file — the 3-band waveforms and a colour-waveform variant.

12.2 The Container Structure

An ANLZ file is a simple, self-describing container. It opens with a header bearing the magic bytes PMAI, then a sequence of length-prefixed tags. Each tag is a four-character identifier followed by its size and its payload. All multi-byte integers are big-endian, and every tag the writer produces is validated against pyrekordbox.

12.4 The Beatgrid Tag — PQTZ

The beatgrid deserves a closer look because it is the tag a DJ feels most directly. PQTZ is a list of entries, each describing one beat:

◈
PQTZ entry layout

Each beat entry is three fields, all big-endian: beat_number (a 16-bit integer, 1–4 within the bar), tempo×100 (a 16-bit integer — the BPM multiplied by 100 so it can be stored as a whole number), and time_ms (a 32-bit integer — the beat's position in milliseconds).

Storing the tempo as BPM×100 is a classic binary-format trick: it preserves two decimal places of precision (128.50 BPM becomes the integer 12850) without needing a floating-point field. Get this scaling wrong and every beat marker on every CDJ drifts.

12.5 The Format Subtleties

The ANLZ format is full of small, unforgiving details. A handful that the writer must get exactly right:

TagThe detail
PCOBA 24-byte header containing an 0xFFFFFFFF sentinel value.
PCO2Its list_type field is a 32-bit integer — not 16-bit, a tempting mistake.
PWV5A header field at offset +20 must hold the exact constant 0x00960305.
PWV6A 20-byte header, with a 32-bit entry_size of 3 before the entry count.
PWV7A 24-byte header, a 32-bit entry_size of 3, and each data entry is the triple [low, mid, high].
▲
There is no error message

When an ANLZ file is wrong, a CDJ does not report a parse error. It simply shows no waveform, or a flat beatgrid, or it silently ignores the file. The only way to know the format is right is to validate it — which is why every tag the writer emits is checked against pyrekordbox, and why the constants above are spelled out explicitly in the code with comments.

12.7 Where ANLZ Files Live

ANLZ files exist in two places, and a shared helper keeps the locations deterministic.

flowchart LR AN["Analysis result"] --> SC["anlz_sidecar
writes locally"] SC --> LOCAL["<music>/.lms_anlz/<hash>/
ANLZ0000.DAT / .EXT / .2EX"] LOCAL --> USB["USB sync copies to"] USB --> STICK["PIONEER/USBANLZ/<bucket>/<hash>/
on the stick"]

app/anlz_sidecar.py is the shared helper. Every track-import path — SoundCloud download, drag-drop import, the folder watcher — calls it, so the resulting .DAT/.EXT/.2EX files always land in a deterministic location next to the audio: <music>/.lms_anlz/<sha-of-path>/. When a USB sync runs, the OneLibrary writer copies them straight to PIONEER/USBANLZ/<bucket>/<hash>/ on the stick. The bucketing keeps any one directory from holding too many entries.

12.8 ANLZ in the Bigger Picture

ANLZ files are one of two binary formats the application reverse-engineers. They describe a single track's analysis. The other format — the PDB database of Chapter 13 — describes the whole library's structure. Together they make a stick a CDJ reads:

Chapter summary

ANLZ files are the per-track binary analysis companions a CDJ reads to draw waveforms and place beat and cue markers. Each track gets three — .DAT, .EXT, .2EX — built from a PMAI header and a sequence of big-endian tags (PPTH, PQTZ, the PWAV/PWV family, PCOB/PCO2). The format is unforgiving — a wrong constant produces no error, just a broken display — so every tag is validated, and a shared sidecar helper keeps the files in a deterministic location ready for USB export.
