Chapter 08 · The Backend

Complete API Reference

All 146 REST routes the backend exposes, grouped by feature. This is the definitive map of the contract between the frontend and the Python sidecar.

8.1 Reading This Reference

Routes are grouped into fifteen feature sections. Each table shows the HTTP method, the path, an authentication marker, and a description. Two markers appear:

MarkerMeaning
AUTHRequires Authorization: Bearer <token>. A missing or wrong token returns 401. Applied to every mutation route.
RLAdditionally rate-limited by a token-bucket — excess requests return 429 with a Retry-After header.

Unmarked routes are open — they are read-only, or the single heartbeat exception. The methods are colour-coded: GET POST PUT PATCH DELETE.

◆
The authentication model in one line

84 of the 85 mutation routes (every POST / PUT / PATCH / DELETE) carry the auth gate. The sole exception is POST /api/system/heartbeat, a body-less liveness probe. Read-only GET routes are loopback-gated only. Chapter 21 explains why.

Feature groups

The shape of the surface tells a story: the USB and SoundCloud groups are the largest, reflecting the two headline workflows; the read-heavy library and playlist groups expose the collection; and a thin system group manages the process lifecycle. Adding a route follows a guided scaffold that applies the auth, lock, and Pydantic-model conventions automatically — described in Chapter 26.

✦
Chapter summary

The backend exposes 146 REST routes across fifteen feature groups. Every mutation route but one carries Bearer-token authentication; the three highest-risk routes add token-bucket rate limiting. Every route returns the uniform {status, data} or {status, message, code} envelope. This catalogue is the contract every frontend feature is built against.
