Chapter 17 · The Frontend

Waveform & State Management

Two pieces of frontend machinery deserve their own chapter: the composed reducer that holds all DAW editing state with undo/redo, and the WaveSurfer-based waveform editor with its 3-band visualisation.

17.1 The DAW State Machine

The DAW of Chapter 16 has a great deal of state — regions, cues, loops, the transport, the selection, the project metadata, the undo history. A coding rule forbids scattering that state across components: all DAW state is managed in audio/; do not duplicate it in component-local state.

The state lives in a single reducer, driven by a useReducer call in DjEditDaw. But a single 1,000-line reducer function would be as unmaintainable as a god-component — so the reducer is composed from five focused sub-reducers.

17.2 The Composed Reducer

audio/DawState.js is a barrel module. It composes five sub-reducers from audio/dawState/ into the public dawReducer and re-exports the shared helpers. Its surface is bit-for-bit identical to the pre-split monolith — the split was a pure refactor.

17.3 Immutability and Undo/Redo

The reducer is strictly immutable — every action returns a new state object, never a mutation of the old one. That discipline is what makes undo/redo simple.

flowchart LR S0["State 0"] -->|"edit"| PUSH1["PUSH_UNDO
snapshot State 0"] PUSH1 --> S1["State 1"] S1 -->|"edit"| PUSH2["PUSH_UNDO
snapshot State 1"] PUSH2 --> S2["State 2"] S2 -->|"UNDO"| S1b["restore State 1"] S1b -->|"REDO"| S2b["restore State 2"]

history.js implements undo/redo as full-state snapshots — a deep clone of the entire state via a JSON round-trip. This is simple and bullet-proof: there is no per-action inverse to get wrong, no risk of a half-undone edit. A restore also clears the selection set, so the user never undoes into a stale selection. The HYDRATE action does the same job for project loading — it resets the whole state to a loaded project.

◆
Why snapshots, not deltas?

A delta-based history (storing each action's inverse) is more memory-efficient but far more error-prone — every action needs a correct inverse, and one wrong inverse corrupts the timeline silently. For an interactive editor where the state is modest in size, a snapshot history trades a little memory for total correctness. It is the right call for a tool a DJ trusts with their cue points.

17.4 The Audio Engine Modules

17.5 The Waveform Editor

Separate from the DAW is the standalone waveform editor — the component used inside the Ranking view for quick, per-track waveform work. Its orchestrator is WaveformEditor.jsx, and it was, before the Slopcode-Cleanup, the single largest file in the frontend at 2,223 lines. It is now a slim orchestrator composing the hooks and sub-components under components/waveform/.

17.6 The 3-Band Waveform

The most distinctive visual in the application is the 3-band waveform — the CDJ-style coloured waveform where bass, mids, and highs are drawn in different colours. It exists in two implementations, one client-side and one native.

Client-side — the band analyzer

utils/AudioBandAnalyzer.js splits a decoded AudioBuffer into three frequency bands using an OfflineAudioContext: a low-pass at 400 Hz, a band-pass from 400 to 2000 Hz, and a high-pass above 2000 Hz. It also generates multi-resolution peak arrays — several levels of detail — so the waveform can be drawn crisply at any zoom level without re-processing.

The layered render

In the waveform editor, useMultibandLayers drives three slave WaveSurfer instances — one per band — and a requestAnimationFrame loop keeps their scroll and playback time locked to the master instance.

The native counterpart — the Rust get_3band_waveform command of Chapter 19 — computes the same split far faster with an FFT. The DAW's track loader prefers the native path and falls back to the client-side band analyzer when it is unavailable. The result is the same: a waveform where a DJ can see the kick, the snare, and the hi-hats at a glance.

17.7 Non-Destructive Preview

One detail ties the waveform editor back to the non-destructive principle. When a user makes a cut, useVisualPreview does not modify the source. Instead it splices a copy of the decoded AudioBuffer in memory and reloads the waveform with the spliced version — debounced, with a generation counter that discards stale results if the user keeps editing. The user sees and hears the edit immediately; the file on disk is untouched until an explicit save. previewBuffer.js keeps an LRU cache of these spliced buffers so repeated previews are fast.

✦
Chapter summary

All DAW editing state lives in one immutable reducer, composed from five focused sub-reducers (regions, transport, selection, cues, history). Undo/redo is implemented as full-state snapshots — simple and correct-by-construction. The standalone waveform editor, once the frontend's largest file, is now a slim orchestrator over a dozen hooks and sub-components built on WaveSurfer.js. Its signature 3-band waveform splits the signal into bass, mids, and highs — client-side via an offline audio context, or natively via the Rust FFT — and previews every cut non-destructively on an in-memory buffer copy.
