SoundCloud Integration

SoundCloud is the application's built-in acquisition source. The integration spans all three layers — a Rust OAuth flow, a Python API client, and a legally-careful Python downloader — and ends in an analysed, imported track.

20.2 The OAuth Flow

Authentication uses OAuth 2.1 with PKCE — the modern, secure flow that never exposes a long-lived secret and never asks the user to type a password into the application. It is implemented in soundcloud_client.rs.

sequenceDiagram participant FE as Frontend participant RS as Rust client participant BR as System browser participant SC as SoundCloud participant BE as Backend FE->>RS: invoke('login_to_soundcloud') RS->>RS: generate code_verifier + code_challenge RS->>BR: open the authorisation URL BR->>SC: user logs in and approves SC->>RS: redirect with ?code=... to the local callback RS->>SC: exchange code + verifier for a token SC-->>RS: access token RS-->>FE: the token string FE->>BE: POST /api/soundcloud/auth-token

The flow has four steps in the Rust client:

  1. get_auth_url() generates a random code_verifier and derives a code_challenge from it (SHA-256, then base64url). It returns the authorisation URL and the verifier.
  2. open::that() opens that URL in the system browser, where the user logs in to SoundCloud directly.
  3. wait_for_callback() spins up a one-shot local HTTP server on a random port and blocks until SoundCloud redirects back to it with the authorisation code.
  4. exchange_code_for_token() POSTs the code and the verifier to SoundCloud's token endpoint and receives the access token.

The token is returned to the frontend as a plain string and forwarded to the backend via POST /api/soundcloud/auth-token — one of only three rate-limited routes in the whole API. The Rust client uses a typed ScError enum for failures and emits sc-login-progress events so the UI can show the stages.

▲
Never log the token

The Rust client's rule is explicit: never log the token, only log token_received: true/false. The backend then stores the token in the OS keyring, not in a file — consistent with the secrets policy of Chapter 21.

20.3 The API Client

soundcloud_api.py talks to SoundCloud's unofficial v2 API. It exposes SoundCloudPlaylistAPI, which fetches the authenticated user's playlists and likes and resolves individual tracks.

Talking to an unofficial API is harder than talking to a documented one, and the client handles three real-world complications:

ComplicationHow the client copes
The client_id is not publishedThe client scrapes a valid client_id dynamically, with a Cloudflare fallback.
Rate limits (HTTP 429)An exponential backoff retry, surfaced as a typed RateLimitError carrying a retry_after.
Large playlists are paginatedFull pagination — the client follows every page to assemble the complete list.

It also defines two typed exceptions — AuthExpiredError, which signals the frontend to re-run the OAuth flow, and RateLimitError — so the route handlers can map them to the AUTH_EXPIRED and RATE_LIMITED error codes of Chapter 7.

20.4 Matching SoundCloud Tracks to the Library

SoundCloudSyncEngine answers a question the user constantly has: "which of these SoundCloud tracks do I already own?" It does so with fuzzy matching.

flowchart LR SCT["SoundCloud track
title + artist + duration"] LIB["Local library tracks"] SCT --> M["Fuzzy match
Jaro-Winkler on title + artist"] LIB --> M M --> TIE["Duration tiebreaker"] TIE --> RESULT["MatchResult

match_to_library() compares each SoundCloud track's title and artist against the local library using Jaro-Winkler string similarity, breaks ties with track duration, and produces a MatchResult with a confidence score. The Preview Matches feature runs this as a dry run, and the Match Inspector modal (Chapter 15) shows the user the per-track confidence before anything is downloaded.

20.5 The Downloader — Two-Stage Acquisition

soundcloud_downloader.py is the most legally careful module in the codebase. Its design goal is to download exactly what the SoundCloud web player itself can play — no more — and to do so through the same signed streams.

flowchart TB START["Download request"] START --> S1{"downloadable = true?"} S1 -->|"yes"| OFF["Stage 1 — official
/tracks/{id}/download"] S1 -->|"no"| S2["Stage 2 — fallback
v2 media.transcodings[]"] S2 --> T1{"transcoding type"} T1 -->|"progressive"| PROG["Direct HTTP stream (MP3)"] T1 -->|"hls"| HLS["HLS segments → ffmpeg -c copy mux"] OFF --> DONE["Audio file in temp"] PROG --> DONE HLS --> DONE

Stage 1 — the official download

When a track's metadata reports downloadable = true, the downloader uses SoundCloud's official /tracks/{id}/download endpoint, which returns the creator's original upload. This is the preferred path.

Stage 2 — the transcodings fallback

When the official download is not offered, the downloader falls back to the v2 API's media.transcodings[] — the same signed CDN streams the SoundCloud web player itself plays. A progressive transcoding is a direct MP3 stream; an hls transcoding is downloaded as segments and remuxed with FFmpeg using -c copy — a repack only, with no re-encode.

The legal gates

The fallback path enforces a set of legal boundaries, all in _resolve_stream_via_transcodings():

GateEffect
Skip snipped: true transcodingsA snipped stream is a 30-second paywall preview — never downloaded as if it were the track.
Honour 401 / 403An access-denied response from the stream-signing endpoint stops the download.
Skip empty transcodings[]A private or removed track yields nothing.
Never probe paid qualityHigh-quality streams are used only when SoundCloud itself returns them — the downloader never probes for Go+ quality.
HLS uses -c copyA repack only — no re-encode, no DRM touched.
▲
Aggressive Download Mode

A hidden setting — revealed only by tapping a muted dot five times — bypasses the default snipped-preview gate. The changelog states its scope plainly: it is intended only for tracks the user has a personal right to. The default behaviour stays inside the same legal surface as the web player.

20.6 The Post-Download Pipeline

A download is not finished when the bytes land. The downloader runs a complete post-processing pipeline so a downloaded track arrives in the library fully prepared.

flowchart TB D1["Dedup gate
already downloaded? reject"] D1 --> D2["Register as 'downloading'
blocks parallel duplicates"] D2 --> D3["Resolve + download by protocol"] D3 --> D4["Move to SoundCloud/<Artist>/<Title>"] D4 --> D5["SHA-256 hash → content dedup"] D5 --> D6["Optional AIFF conversion (ffmpeg)"] D6 --> D7["Apply metadata + cover art"] D7 --> D8["Register as 'downloaded'"] D8 --> D9["Background: analyse → import → SC_ playlist"]

The notable steps:

  • Two deduplication gates. Before downloading, an O(1) check against the registry by SoundCloud track ID; after downloading, a SHA-256 content-hash check that catches the same audio arriving by a different route.
  • A 'downloading' registry entry is written immediately, so two parallel requests for the same track cannot both proceed.
  • Optional AIFF conversion via FFmpeg, controlled by the sc_download_format setting.
  • Metadata application — title, artist, album, genre, year, comment, and a resized cover image are written into the file's tags.
  • A background pipeline then analyses the track, imports it into the library, and auto-sorts it into a playlist named SC_<playlist>.

20.7 The Download Registry

The registry that backs deduplication was introduced in Chapter 10. From the SoundCloud feature's point of view it provides four things: an O(1) already-downloaded check, content-hash deduplication, a paginated history log with aggregate statistics, and a stable per-machine device ID so a DJ with several computers can see which machine downloaded what. The registry entry also moves through statuses — downloading → downloaded, or failed — which the Download Manager view turns into a per-task stage timeline.

Chapter summary

SoundCloud integration spans three layers: a Rust OAuth 2.1 + PKCE flow that never exposes a password, a Python API client that copes with an unofficial API (dynamic client-id scraping, backoff, pagination) and fuzzy-matches tracks to the library, and a legally careful downloader that acquires audio in two stages — the official endpoint, then the same signed streams the web player uses — behind explicit legal gates. A full post-download pipeline analyses, tags, imports, and auto-playlists every track, backed by a two-axis deduplication registry.
