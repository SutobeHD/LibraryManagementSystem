Chapter 14 · The Frontend

Frontend Architecture

The frontend is a React 18 single-page application of over a hundred components — lazy-loaded, error-bounded, and talking to the backend through a single hardened HTTP client.

14.1 The Frontend in Outline

Everything the user sees is React, rendered inside the Tauri WebView. The frontend is roughly 27,000 lines of JavaScript across 111 source files — deliberately plain JavaScript, not TypeScript, with a jsconfig.json giving the editor type hints. Vite builds and serves it; Tailwind styles it.

14.2 The Source Tree

The frontend source is organised by role, not by feature — views, the audio engine, the API layer, and shared state each have a home.

flowchart TB SRC["frontend/src/"] SRC --> MAIN["main.jsx — app root, router, sidebar"] SRC --> API["api/ — the axios client"] SRC --> STORE["store/ — auth state"] SRC --> AUDIO["audio/ — the DAW state machine"] SRC --> COMP["components/ — views + shared UI"] COMP --> CDAW["daw/ — the DJ Edit DAW"] COMP --> CED["editor/ — the non-destructive editor"] COMP --> CWAVE["waveform/ — the waveform editor"] COMP --> CSET["settings/ — settings tabs"] COMP --> CUSB["usb/ — USB panels"] SRC --> UTIL["utils/ — logger, band analyzer"] SRC --> CONF["config/ — constants"]

14.3 The Application Root

main.jsx is the root. It is responsible for five things:

  • The sidebar — a 220-pixel navigation rail grouping the views into Library, Editor, Sync, Utilities, and Lab.
  • The mode picker — the startup screen where the user chooses Live, Standalone, Import, or a Defined Path.
  • The tab router — a lazy-loaded router that swaps the active view.
  • A top-level error boundary so one view's crash cannot white-screen the whole application.
  • Lifecycle polling — the heartbeat and library-status checks — and the session-token bootstrap.

14.4 Lazy Loading and Error Isolation

The application has many large views. Loading all of them upfront would make startup slow. Instead, every feature view is lazy()-imported and rendered inside a <Suspense> boundary with a per-tab ErrorBoundary.

flowchart TB ROUTER["Tab router (main.jsx)"] ROUTER --> SUS["Suspense boundary"] SUS --> EB["per-tab ErrorBoundary"] EB --> LV["lazy() view
loaded only when first opened"] EB -.->|"on crash"| FALLBACK["Error fallback UI
the rest of the app stays alive"]

The benefit is twofold: a fast first paint, because only the active view's code is downloaded, and resilience, because a decode error in the waveform editor or a render bug in the USB view degrades to a recoverable error panel rather than a blank window.

14.5 The API Client

The single most important frontend module is api/api.js — the shared axios instance. A coding rule states it plainly: always import from here, never use raw fetch(). The cleanup era found and removed five raw fetch() calls; the client is now the only HTTP path.

The client is far more than a base URL. It carries:

CapabilityWhat it does
Auth bootstrapAt module load, races two ways to obtain the session token, then resolves a promise every request waits on.
Bearer injectionA request interceptor attaches Authorization: Bearer <token> to every call.
401 refresh queueOn a 401, requests queue behind a single re-auth attempt rather than each retrying in parallel.
429 backoffA rate-limited response triggers an exponential backoff retry.
TimeoutsA 10-second default, disabled (timeout: 0) for long-running calls such as USB sync.
Tauri detectionDetects whether it is running inside Tauri or a plain browser, and adapts the token path accordingly.

The bootstrap race

The frontend cannot make an authenticated request until it has the session token, and where the token comes from depends on the runtime:

flowchart TB START["api.js loads"] --> Q{"Running in Tauri?"} Q -->|"yes"| IPC["invoke('get_session_token')
retry loop until ready"] Q -->|"no — browser dev"| DEV["fetch('/dev-token')
Vite middleware reads the token file"] IPC --> SET["setSessionToken(token)"] DEV --> SET SET --> READY["bootstrap promise resolves
queued requests proceed"] IPC -.->|"both fail"| FAIL["authBootstrapFailed = true
persistent toast, mutation UI disables"] DEV -.->|"both fail"| FAIL

If both paths fail — a corrupted token file, an unregistered IPC handler, a fresh dev clone with no sidecar — the client flips an authBootstrapFailed flag and shows a persistent toast. Mutation-capable UI is expected to disable itself when this flag is set, so the user is not invited to fire writes that can only 401; read-only views stay usable.

14.6 Shared Auth State

store/authStore.js is a tiny module-level store — not a heavy state library, just a handful of functions sharing the auth state across the app: the current token, the bootstrap-failed flag, and the bootstrap promise itself (stashed so a view awaiting the initial token does not re-trigger the IPC handshake). It is a good example of the frontend's preference for the smallest tool that does the job.

14.7 State Management Philosophy

The frontend has no global state framework. State is handled at the level where it belongs:

Kind of stateWhere it lives
The DAW's editing stateA useReducer driving a composed reducer (Chapter 17). Component-local duplication of this is forbidden.
Auth token + flagsThe tiny authStore module.
Cross-cutting notificationsA React context — the toast provider.
View-local UI statePlain useState in the component.
Server dataFetched on demand through the API client; not globally cached.

14.8 The Melodex Design System

The visual identity is a design system the project calls Melodex, configured in tailwind.config.js and index.css. Its signature is the amber accent — #E8A42A — over a near-black surface palette.

Token familyPurpose
mx-*Surfaces — panels, cards, backgrounds.
ink-*Text colours — from bright to dim.
line-*Borders and dividers.
amber2.*The accent palette.

The typefaces are DM Sans for text and JetBrains Mono for code and numbers. index.css defines a set of reusable primitives — .btn-primary, .btn-ghost, .input-glass, .glass-panel, .mx-card, .mx-chip — so components compose a consistent look from shared building blocks. This very handbook borrows the Melodex palette.

14.9 Frontend Conventions

The Slopcode-Cleanup left the frontend with a strict rulebook, still enforced:

RuleReplaces
No alert()/confirm()/prompt()A toast system and promise-based ConfirmModal / PromptModal.
No raw fetch()The shared axios client, or invoke() for Tauri.
No stray console.logutils/log.js — log.debug/log.info are silenced in production; log.warn/log.error always pass.
No empty catch {}Every catch logs and, where relevant, toasts.
No magic numbersconfig/constants.js — heartbeat intervals, timeouts, durations.

Every data-fetching component is also required to handle loading and error states explicitly — a loading spinner and an error message are mandatory, not optional.

Chapter summary

The frontend is a 100-plus-component React 18 SPA, lazy-loaded and error-bounded so it starts fast and fails gracefully. Every HTTP call goes through one hardened axios client with an auth-token bootstrap, a 401 refresh queue, and 429 backoff. State lives where it belongs — a reducer for the DAW, a tiny store for auth, context for toasts, local state for everything else. The Melodex amber design system and a strict post-cleanup rulebook keep it consistent.
