Chapter 19 · Native & Integrations

The Native Audio Engine

The audio/ module is where Rust earns its place in the stack — memory-mapped decoding, a lock-free playback pipeline, FFT-based waveform analysis, key detection, and acoustic fingerprinting.

19.1 Why This Is Rust

Audio playback has a hard real-time constraint. The operating system asks the application for the next buffer of samples on a strict schedule, and if the application is even slightly late — because a garbage collector paused, or a lock was contended — the user hears a click, a pop, or a dropout. A garbage-collected language cannot make the guarantee; Rust, with no GC and precise control over memory and threads, can.

So the native audio engine lives in src-tauri/src/audio/ — seven Rust files covering decoding, playback, analysis, export, metadata, and fingerprinting.

19.3 Decoding — engine.rs

engine.rs defines AudioEngine and AudioController, responsible for getting audio off disk and into decoded samples. It does two clever things:

  • Memory-mapped loading. Instead of reading the whole file into a buffer, it memory-maps it with memmap2 — the operating system pages the file in on demand, and there is no upfront copy. This is why playback can start in under 50 milliseconds even for a large FLAC file.
  • Format-agnostic decoding. It uses Symphonia, a pure-Rust decoding library, behind a decoder abstraction that handles MP3, FLAC, WAV, ALAC, and ISO-MP4 uniformly. The decoder is selected from the file's container, and the channel count is read from the decoder — never hardcoded.
◆
No hardcoded stereo

A coding rule, fixed during the Slopcode-Cleanup, forbids assuming two channels. The audio code reads the channel count from the decoder and either fails fast on an unexpected layout or downmixes deliberately — a mono file or a multichannel file is handled honestly rather than producing garbled output.

19.4 Playback — playback.rs

playback.rs defines PlaybackEngine, which drives the actual sound output through CPAL — a cross-platform library that abstracts the operating system's audio device.

The challenge here is threading. CPAL runs the audio callback on its own high-priority thread, and that callback must never block — no locks, no allocation, no I/O. Yet the rest of the application needs to control playback (play, pause, stop) and feed it samples. The engine solves this with a clean separation:

flowchart TB subgraph App["Application threads"] CTRL["Control — play / pause / stop"] FEED["Sample feeder"] end subgraph Audio["The audio thread (high priority)"] CB["CPAL callback
never blocks"] end FEED -->|"ringbuf — lock-free push"| CB CTRL -->|"Arc<AtomicBool> flags
is_playing · shutdown"| CB CB --> DEV(["Output device"])

Samples cross the thread boundary through a ringbuf lock-free ring buffer — the feeder calls try_push, the callback calls try_pop, and neither ever blocks. Control crosses through Arc<AtomicBool> flags — is_playing, shutdown — which the callback reads without locking.

▲
The unsafe impl Send + Sync that was removed

A CPAL Stream is !Send on Windows by design. An earlier version of the code papered over this with a naked unsafe impl Send + Sync for PlaybackEngine — a genuine soundness hazard. The Slopcode-Cleanup refactored it properly: the Stream is now built, played, and dropped entirely inside the dedicated audio thread, so it never crosses a thread boundary. PlaybackEngine is Send + Sync through the compiler's own derivation, with no unsafe at all. The engine also recovers from a DeviceNotAvailable error by emitting a frontend event rather than panicking.

19.5 Analysis — analysis.rs

analysis.rs is the native, low-latency analysis path. It does not replace the Python engine of Chapter 11 — Python remains the accuracy reference — but it lets the native side draw a waveform and estimate a key without a round-trip to the backend.

The 3-band waveform

compute_waveform() uses RustFFT to transform sliding frames of the signal, then splits the energy into three frequency bands. The boundaries are defined in Hz, not FFT bin indices:

BandRangeHears
Low20 Hz – 250 HzBass, the kick drum.
Mid250 Hz – 4 kHzVocals, the snare, most melodic content.
High4 kHz – 20 kHzHi-hats, cymbals, air and presence.
◈
Hz, not bin indices — a real bug, fixed

An FFT produces "bins", and which bin corresponds to which frequency depends on the sample rate. The original code hardcoded bin indices, so the 3-band split was wrong for any file not at 44.1 kHz. The fix introduced hz_to_bin(hz, frame_size, sample_rate) — every band boundary is now converted from Hz to a bin at the file's actual sample rate. The rule is explicit: never hardcode bin indices.

Key detection

detect_key() implements the Krumhansl-Schmuckler algorithm. It computes a chromagram — the energy in each of the twelve pitch classes — over Hann-windowed FFT frames, L2-normalises it, and Pearson-correlates it against all 24 rotations of the standard major and minor key profiles. The best-correlating rotation is the key, returned in Camelot notation. An earlier version of this function simply returned the hardcoded string "8A" for everything — the cleanup replaced that fake value with the real algorithm.

19.6 Offline Render — export.rs

export.rs handles offline audio synthesis — rendering a project's timeline of regions, fades, and gains into a finished WAV via the hound crate. It backs the WAV path of the DAW's export modal (Chapter 16). The cleanup hardened it against two crashes: an invalid region where the end precedes the start is now rejected, and a sample-count calculation that could overflow is now bounds-checked.

19.7 Metadata — metadata.rs

metadata.rs reads and writes embedded audio tags through the lofty crate — ID3 for MP3, FLAC tags, ALAC and AIFF metadata. It is the native counterpart to the Python audio_tags module; the native side uses it where it already has the file open for playback or analysis.

19.8 Acoustic Fingerprinting — fingerprint.rs

fingerprint.rs is a Chromaprint-style acoustic fingerprinter, the engine behind the duplicate finder. It reduces an audio file to a compact sequence of hash words that can be compared even when two files differ in format, bitrate, or volume.

flowchart TB D["Decode via Symphonia
→ mono f32 samples"] D --> RS["Resample to 11025 Hz"] RS --> FR["Sliding 128 ms frames
50% overlap"] FR --> MEL["32-band Mel spectrogram
via Goertzel"] MEL --> GRAD["Temporal gradient
→ 1 bit per band per frame"] GRAD --> WORDS["Pack into u32 hash words"] WORDS --> SIM["hamming_similarity()
→ 0.0 ... 1.0"]

The algorithm decodes the file to mono, resamples it to 11025 Hz, runs a 32-band Mel spectrogram over 128-millisecond half-overlapping frames, takes the temporal gradient of each band's energy to get one bit per band per frame, and packs those bits into u32 hash words. hamming_similarity() then compares two fingerprints by Hamming distance, yielding a similarity from 0 to 1. Two commands expose it — fingerprint_track for one file and fingerprint_batch for many, the batch version emitting fingerprint_progress events.

◆
A local fingerprint, not an external one

This fingerprint is designed for finding duplicates within a user's own library — it is not bit-compatible with the AcoustID service. The roadmap (Chapter 25) proposes a two-tier design that keeps this Rust fingerprinter for local clustering and adds an external Chromaprint path for online metadata lookup.

19.9 The Shared State

The audio commands of Chapter 18 operate on a shared AudioCommandState — an Arc<Mutex<AudioEngine>> and an Arc<Mutex<PlaybackEngine>> registered with Tauri's .manage(). A command locks the engine it needs, does its work, and maps any lock-poisoning or error to a String the frontend can display — never a panic.

✦
Chapter summary

The native audio engine is Rust because audio playback has a hard real-time deadline a garbage-collected language cannot meet. It memory-maps files and decodes them with Symphonia; it plays them through a CPAL callback fed by a lock-free ring buffer and steered by atomic flags — with no unsafe after a deliberate refactor. It draws sample-rate-correct 3-band waveforms with RustFFT, detects key by pitch-class-profile correlation, renders offline exports, reads tags, and fingerprints audio for the duplicate finder.
