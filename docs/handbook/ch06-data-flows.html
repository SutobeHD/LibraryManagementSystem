Chapter 06 · Architecture

Data Flows End-to-End

Eight journeys traced through the running system — from a user click to the bytes on disk. If Chapter 4 is the map, this chapter is the set of routes drawn on it.

6.1 Why Trace Flows

A component diagram tells you what the parts are. A data flow tells you how they cooperate to get one job done. The eight flows in this chapter are the ones that recur most in the rest of the handbook; seeing them whole now makes the deep-dive chapters easier to read. Each flow is presented as a sequence diagram followed by a short narration.

6.2 Flow 1 — Loading the Library

The first thing the application does after boot is show the user their tracks.

sequenceDiagram participant U as User participant FE as Frontend participant BE as Backend participant DB as Library (XML / master.db) U->>FE: opens the app FE->>BE: GET /api/library/tracks BE->>DB: read tracks (active mode) DB-->>BE: raw track rows BE->>BE: normalise to track dicts BE-->>FE: [ {id, title, artist, bpm, key, path, ...} ] FE->>FE: TrackTable renders the sortable list

The cost of this flow is paid once. In XML mode the file is parsed into an in-memory dictionary cache on first load; every subsequent read is a sub-five-millisecond cache hit. The frontend's TrackTable component then renders the result with Camelot-coloured key cells. The library layer behind DB is Chapter 10.

6.3 Flow 2 — Native Audio Playback

When the user clicks a track to play it, the request does not go to the Python backend. It goes straight to the Rust audio engine over IPC, because playback latency must be measured in milliseconds.

sequenceDiagram participant U as User participant FE as Frontend participant RS as Rust Audio Core participant SPK as Output Device U->>FE: clicks a track FE->>RS: invoke('load_audio', {path}) RS->>RS: validate path, memory-map the file RS->>RS: Symphonia decoder selected by format RS->>SPK: PlaybackEngine opens a CPAL stream RS-->>FE: AudioInfo (duration, sample rate, channels) RS->>SPK: audio thread feeds the ring buffer RS-->>FE: progress events

The native audio engine is Chapter 19. The key point here is the routing decision: realtime concerns take the IPC channel to Rust, never the HTTP channel to Python.

6.4 Flow 3 — Waveform Computation

Drawing a track's waveform is a heavy numerical job, so it too is handed to Rust, which has the FFT crate and the speed.

sequenceDiagram participant FE as Frontend participant RS as Rust Audio Core FE->>RS: invoke('get_3band_waveform', {path}) RS->>RS: decode samples via Symphonia RS->>RS: RustFFT over sliding frames RS->>RS: split energy into low / mid / high bands RS-->>FE: { low[], mid[], high[], peaks[] } FE->>FE: TimelineCanvas paints the 3-band waveform

The three bands — roughly bass and kick, mids and snare, hi-hats and air — are computed by converting Hz boundaries to FFT bins at the file's actual sample rate. Chapter 19 explains why hardcoding bin indices was a bug worth fixing.

6.5 Flow 4 — Background BPM & Key Analysis

Full analysis — tempo, key, beatgrid — is too slow to block a request, so it runs in a worker process and the frontend polls for the result.

sequenceDiagram participant FE as Frontend participant BE as Backend participant PP as Process Pool FE->>BE: POST /api/audio/analyze {path} BE->>PP: submit task to ProcessPoolExecutor BE-->>FE: { task_id } loop until done FE->>BE: GET /api/audio/analyze/{task_id} BE-->>FE: { status: "pending" } end PP->>PP: madmom beat tracking + essentia key PP-->>BE: { bpm, key, confidence, beatgrid } FE->>BE: GET /api/audio/analyze/{task_id} BE-->>FE: { status: "done", bpm, key, ... }

This submit-and-poll pattern recurs across the application — for SoundCloud downloads, USB syncs, and duplicate scans. It keeps the API responsive no matter how long the real work takes. The analysis engine is Chapter 11.

6.6 Flow 5 — Non-Destructive Editing

When the user edits cue points, the beatgrid, or a volume envelope in the DAW, the source audio file is never touched. The edit lives in the frontend's state until the user saves a project.

sequenceDiagram participant U as User participant ST as DAW State (reducer) participant CV as Timeline Canvas participant BE as Backend participant F as .rbep project file U->>ST: edits a cue / beatgrid / envelope ST->>ST: dawReducer — pure state update ST->>CV: canvas re-renders (no file write) U->>BE: POST /api/projects/save BE->>F: serialise overlay to .rbep Note over F: the source audio file is never modified

The .rbep file is an overlay — it records cues, beatgrids, and envelopes by reference to the source file, never a copy of the audio. The DAW and its reducer are Chapters 16 and 17.

6.7 Flow 6 — SoundCloud Sync & Download

The SoundCloud flow is the longest in the application because it spans all three active layers — the Rust OAuth flow, the Python API client, and the React UI — and ends in a downloaded, analysed, imported track.

sequenceDiagram participant U as User participant FE as Frontend participant RS as Rust Core participant SC as SoundCloud participant BE as Backend U->>FE: "Connect SoundCloud" FE->>RS: invoke('login_to_soundcloud') RS->>SC: PKCE OAuth in the system browser SC-->>RS: authorization code (local callback) RS->>SC: exchange code for token RS-->>FE: access token FE->>BE: POST /api/soundcloud/auth-token U->>FE: picks a playlist, clicks Download FE->>BE: POST /api/soundcloud/download BE->>SC: resolve stream (official API, then fallback) BE->>BE: download, hash, dedup-check BE->>BE: analyse, import, auto-playlist BE-->>FE: task progress until complete

The OAuth step is in Rust because it needs to open a real browser and run a one-shot local HTTP server to catch the callback — host-process capabilities the WebView does not have. Everything after the token is Python. The whole integration is Chapter 20.

6.8 Flow 7 — USB Sync

The USB sync is where the application earns its name as a Rekordbox competitor: it writes a stick a CDJ-3000 reads natively.

sequenceDiagram participant U as User participant FE as Frontend participant BE as Backend participant USB as USB Stick U->>FE: selects a drive + playlists FE->>BE: GET /api/usb/diff/{device_id} BE-->>FE: what would change U->>FE: confirms the sync FE->>BE: POST /api/usb/sync BE->>USB: acquire the sync lock file BE->>USB: write exportLibrary.db (OneLibrary) BE->>USB: write export.pdb / exportExt.pdb (legacy) BE->>USB: copy audio + ANLZ + artwork BE->>USB: seed hardware-settings files BE-->>FE: streamed stage / progress events

This flow writes the most demanding data in the whole application — the byte-exact Pioneer database formats. A single wrong flag silently corrupts the stick. Chapters 12 and 13 document those formats with the care they require.

6.9 Flow 8 — The Authentication Bootstrap

The final flow is invisible to the user but underpins every other one: how the frontend obtains the token that authorises its requests.

sequenceDiagram participant P as Python Sidecar participant T as Tauri Shell participant FE as Frontend participant API as FastAPI route P->>P: SESSION_TOKEN = token_urlsafe(32) P->>T: stdout "LMS_TOKEN=<value>" T->>T: capture token, drop the line from logs P->>P: also persist token to user-data file FE->>T: invoke('get_session_token') T-->>FE: the captured token FE->>FE: axios interceptor stores it FE->>API: every request carries Authorization: Bearer API->>API: require_session compares constant-time API-->>FE: 200, or 401 on any mismatch

In browser-development mode the token handoff differs slightly: the Vite dev server reads the on-disk token file and re-exposes it at a /dev-token endpoint, since there is no Rust shell to capture the stdout banner. Either way, no request reaches a mutation route without a valid Bearer token. Chapter 21 is the full security model.

6.10 The Flows on One Map

Stepping back, the eight flows divide cleanly by which channel and which layer they exercise:

FlowPrimary channelHeavy lifting in
1 · Library loadHTTPPython — database layer
2 · PlaybackIPCRust — CPAL + Symphonia
3 · WaveformIPCRust — RustFFT
4 · AnalysisHTTPPython — process pool
5 · Editinglocal / HTTP on saveReact — the reducer
6 · SoundCloudIPC then HTTPRust (OAuth) + Python (download)
7 · USB syncHTTPPython — the USB writers
8 · Auth bootstrapstdout + IPCPython (mint) + Rust (capture)

The recurring shape is now clear: the frontend orchestrates, the channel is chosen by the need (latency and OS access push work to Rust; data and orchestration go to Python), and anything slow is made asynchronous with a task ID and polling.

✦
Chapter summary

Eight traced flows reveal the system's working rhythm: library load and analysis go over HTTP to Python; playback and waveform go over IPC to Rust; editing is pure frontend state until an explicit save; SoundCloud spans Rust OAuth and Python download; USB sync writes byte-exact Pioneer formats; and the auth bootstrap quietly secures everything else. Slow work is always pushed to a background worker behind a polled task ID.
