Chapter 13 · The Backend

Pioneer USB Export

Writing a USB stick that a CDJ-3000 reads natively is the capability that makes this a real Rekordbox competitor. It is also the most exacting code in the project — two reverse-engineered database formats, verified byte-for-byte against a real Pioneer drive.

13.1 The Stakes

Every other feature in the application has a forgiving failure mode. Get a waveform colour slightly wrong and the user sees a slightly-off waveform. Get the USB export wrong and the CDJ shows a single dialog — "Device library is corrupted" — and refuses to load anything. There is no partial success and no diagnostic. The stick either works completely or not at all.

That is why this subsystem is held to a unique standard: its byte layout is verified byte-for-byte against a real Pioneer-exported USB drive, and a regression test guards every structural invariant.

13.2 The Anatomy of a CDJ-Ready Stick

A USB stick a CDJ can play has a specific directory structure under a PIONEER/ folder. The sync engine writes all of it.

13.3 Two Database Formats

The stick carries the library in two formats, because Pioneer hardware spans two generations of database design.

FormatFile(s)Read by
OneLibraryrekordbox/exportLibrary.dbRekordbox 6/7 — lets the desktop software auto-detect the stick.
Legacy DeviceLibrary (PDB)export.pdb, exportExt.pdbOlder CDJ firmware that predates the OneLibrary format.

Each format has its own writer module. They are among the hardest files in the codebase, for different reasons.

13.4 The OneLibrary Writer — Working Around a Broken Library

app/usb_one_library.py writes exportLibrary.db. The obvious approach would be to call pyrekordbox's OneLibrary.create() and populate it. That approach does not work: in version 0.1.7, create_content() raises "Unexpected null for non-null column" on every call, and there is no Python-level constructor for a new content row.

The solution is a template-based approach:

flowchart LR TPL["app/templates/
exportLibrary_template.db
16 placeholder rows"] TPL -->|"copy onto the stick"| STICK["exportLibrary.db
on the USB"] STICK -->|"update_content()
mutate placeholders"| FILLED["Populated with
real track data"]

A bundled, anonymised template database — derived from a real Rekordbox-exported stick — ships with placeholder content rows. The writer copies it onto the USB stick and uses update_content() (which does work) to populate the placeholders with real track data. The other creation calls — for images, artists, albums, genres, keys, labels, playlists — work normally and are used as-is.

▲
Two consequences of the template approach

First, the template's placeholder count is a hard ceiling on track count — a larger template raises the cap. Second, after writing, the pyrekordbox 0.1.7 handle exposes no close() or commit(), so the writer deliberately drops the handle, forces garbage collection, and reopens the database to flush the SQLite write-ahead log — otherwise up to 400 KB of unmerged WAL frames are left on the stick. A dedicated test, test_onelibrary_wal_flush.py, verifies the WAL is empty afterwards.

13.5 The PDB Writer — A Format Built From Scratch

app/usb_pdb.py writes the legacy export.pdb and exportExt.pdb files. There is no library that does this — the module implements the format from scratch, following the community "Crate Digger" reverse-engineering of Pioneer's DeviceSQL database.

The page structure

A PDB file is a flat sequence of 4 KiB pages. A file header points to a directory of 20 tables; each table is a chain of pages; each data page holds rows.

flowchart TB HDR["File header
points to the table directory"] HDR --> DIR["20-table directory
tracks, artists, albums, genres,
keys, colours, playlists, ..."] DIR --> PG["4 KiB pages per table"] subgraph Page["One 4 KiB data page"] PH["Page header
row count, flags, accounting"] ROWS["Row data — grows from the top"] IDX["Row index — grows from the bottom
reverse order, bit-packed"] end PG --> Page

Within a page, row data grows downward from the header while a bit-packed row index grows upward from the end of the page — a classic space-efficient slotted-page design. The module also implements a DeviceSQL string encoder: short ASCII strings and long strings (encoded UTF-16-LE) use different on-disk forms.

The row encoders

Each of the 20 tables has its own row encoder, and each was verified byte-identical to samples lifted from a real Pioneer F:-drive:

TableRow content
Tracks (djmdContent)A fixed 0x88-byte header plus 21 string offsets.
Artists / Albums"Far-name" rows with compact structured headers.
Genres / Labels / Keys / ColoursSmall rows — an ID plus a DeviceSQL string.
Playlist tree / Playlist entriesThe folder-and-playlist structure and its membership.
ArtworkAn ID plus the DeviceSQL artwork path.

The anti-corruption invariants

The comments in usb_pdb.py enumerate ten distinct fixes, each discovered by comparing the writer's output to a real Pioneer drive byte-by-byte. A representative sample of the invariants that, if violated, produce the dreaded "corrupted" dialog:

InvariantWhy it matters
Data-page flag 0x34Track data pages use flag 0x34 (not 0x24). The wrong flag corrupts the F:-drive parity.
Per-table blank pagesEach of the 20 tables needs its own dedicated all-zero blank page. Sharing one global blank flags the library corrupted.
Chain terminatorThe terminating next_page is patched from 0 to the table's own blank page — so the CDJ does not follow the chain into the file header.
The free_size formulaPage free-space accounting uses the effective footer size, not the physical block size — Rekordbox checks the arithmetic.
Empty tablesAn empty table is a single index page where first equals last.
◈
"Read the file before changing it"

The project's coding rules single this file out: every magic number is explained in a comment, the layout is byte-verified, and tests/test_pdb_structure.py guards every structural invariant. A failing PDB test means the change is wrong — not that the test needs updating. The PDB writer handles libraries up to roughly 500 tracks; larger libraries fall through to the legacy XML path.

13.6 The Sync Engine

app/usb_manager.py ties it all together with three classes.

ClassResponsibility
UsbDetectorScan drives, identify genuine removable USB sticks, and initialise new ones.
UsbProfileManagerPersistent CRUD for sync profiles — which playlists go to which stick — and global USB settings.
UsbSyncEngineThe actual sync — diff calculation, collection and playlist sync, metadata sync.
sequenceDiagram participant U as User participant SE as UsbSyncEngine participant USB as USB Stick U->>SE: start sync (drive, playlists) SE->>USB: acquire .rbep_sync_lock SE->>SE: calculate incremental diff SE->>USB: write exportLibrary.db (OneLibrary) SE->>USB: write export.pdb / exportExt.pdb SE->>USB: copy audio, ANLZ, artwork SE->>USB: seed MYSETTING files SE->>USB: release the lock SE-->>U: stream { stage, message, progress }

The sync engine is built defensively, because real USB sticks misbehave:

  • A lock file (.rbep_sync_lock) prevents two syncs from corrupting one stick.
  • An incremental diff means only changed tracks are copied — a re-sync of a large library is fast.
  • Filename safety — _clean_filename() strips trailing dots and spaces and rejects Windows reserved names (CON, PRN, NUL...), because Windows silently drops trailing dots at directory-creation time and later writes then fail.
  • XML safety — _xml_safe() strips ASCII control characters from metadata so the exported XML stays parseable.
  • Copy-error classification — only a real disconnect (the drive root vanishing) aborts the batch; a single bad filename is logged and skipped, so one bad track does not kill a 3,000-track sync.

13.7 Artwork and Hardware Settings

Bucketed artwork

app/usb_artwork.py extracts embedded cover art — ID3 APIC, FLAC Picture, MP4 covr atoms, or a cover.jpg sidecar — and resizes it via Pillow to two JPEGs: a small (~80×80) and a medium (~500×500). They are written to a bucketed PIONEER/Artwork/<bucket>/ path so legacy CDJs see the same cover art as Rekordbox 7.

Hardware settings

app/usb_mysettings.py writes the CDJ and DJM hardware-settings files — MYSETTING.DAT, MYSETTING2.DAT, DJMMYSETTING.DAT — covering 42 editable fields across the player, the extended player, and the mixer. It exposes a JSON schema so the UI can build the right dropdowns without hardcoding any enum tables, and the settings are auto-seeded with sensible defaults on every sync.

13.8 Play-Count Synchronisation

A DJ plays the stick at a gig; the CDJ increments play counts on the stick; the PC library is now out of date. app/playcount_sync.py reconciles the two with a three-way diff.

flowchart LR PC["PC play counts"] --> DIFF{"Three-way diff
vs last-sync snapshot"} USB["USB play counts"] --> DIFF SNAP["Last-sync snapshot"] --> DIFF DIFF --> AUTO["Auto-resolved
only one side changed"] DIFF --> CONF["Conflicts
both sides changed"] CONF --> RES["User picks a strategy
take max / take PC / take USB / sum"]

Where only one side changed, the engine resolves it automatically. Where both changed, it surfaces a conflict and lets the user choose a strategy. The result is committed back to both the PC database and the USB XML.

13.9 Why the Stick Can Carry Both Formats Safely

A subtlety worth noting: writing both the OneLibrary database and the legacy PDB onto one stick can itself trigger the "corrupted" dialog on Rekordbox 7 if the still-imperfect PDB is present. The sync engine therefore treats the legacy PDB writer as opt-in — a profile setting controls it, and when it is off, any stale export.pdb already on the stick is deleted. The OneLibrary path is the safe default; the PDB path is there for older hardware that needs it.

Chapter summary

USB export writes a stick a CDJ reads natively, in two reverse-engineered database formats. The OneLibrary writer works around a broken pyrekordbox creation path with a bundled template and a deliberate WAL flush. The PDB writer implements the DeviceSQL format from scratch — 4 KiB slotted pages, a 20-table directory, ten byte-verified anti-corruption invariants. The sync engine adds a lock file, an incremental diff, and hard-won filename and copy-error defences. Artwork, hardware settings, and three-way play-count sync round out a stick that just works in the booth.
