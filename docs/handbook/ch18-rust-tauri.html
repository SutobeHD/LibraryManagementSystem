Chapter 18 · Native & Integrations

The Rust / Tauri Layer

The Rust layer is the application's true host process. It creates the window, supervises the Python sidecar, captures the session token, and exposes a tight set of native capabilities to the frontend over Tauri's IPC bridge.

18.1 The Role of the Rust Layer

When the user launches the application, the program that actually starts is the Rust/Tauri binary. Everything else — the WebView, the Python sidecar — is something the Rust process creates and owns. The Rust layer is small, roughly 2,900 lines across ten files, but it is the load-bearing host.

It has four distinct jobs:

18.3 The Application Entry Point

main.rs is where the Tauri application is built and run. At startup it:

  1. Initialises the logger (env_logger at info level).
  2. Creates the splashscreen window.
  3. Spawns the Python sidecar and starts the token-capture reader.
  4. Registers the shared state objects with Tauri's .manage() — the audio command state and the session-token holder.
  5. Registers the IPC command handlers.
  6. Creates the main window and loads the frontend.

It also installs a RunEvent::Exit handler that kills the sidecar child process when the application closes — so there are never orphan Python processes left behind.

18.4 The IPC Bridge

The frontend reaches the Rust layer through Tauri's IPC: the JavaScript side calls invoke('command_name', args) and receives a Result; the Rust side can also push events the frontend subscribes to with listen().

◆
Commands return results, events report progress

The two halves of the IPC bridge play different roles. A command is a request-response call — the frontend awaits a Result. An event is a one-way push — the Rust side emits progress as a long job runs. A long operation like an export uses both: a command to start it, then a stream of events to report it.

18.5 The Sidecar Supervisor

One of the Rust layer's most important jobs is supervising the Python sidecar. It spawns the process, reads its output, and — crucially — captures the session token from that output.

sequenceDiagram participant M as main.rs participant P as Python sidecar participant S as SessionToken state participant L as Log channel M->>P: spawn sidecar process loop every stdout line P-->>M: a line of output alt line starts with "LMS_TOKEN=" M->>S: store the token (strip prefix) Note over M,L: line is dropped — never logged else any other line M->>L: forward to the log channel end end

The Python sidecar emits its session token as its first line of standard output, in the form LMS_TOKEN=<value>. The supervisor's reader thread inspects every line: the first one matching that prefix is stripped, the token stored in a shared Mutex<String>, and the line dropped from the forwarded log stream so the token never reaches a log file. Every other line is forwarded normally. The frontend later retrieves the stored token via the get_session_token command. This same capture logic runs on both spawn paths — the production shell-sidecar spawn and the development child spawn. Chapter 21 explains why this matters.

18.6 The Splashscreen Lifecycle

The splashscreen exists because the sidecar takes a moment to boot. The Rust layer shows the splash first, and the main window is not revealed until the frontend — having confirmed the backend is alive — calls close_splashscreen. There is a brief, expected flash of the splash window before the main window takes over; the e2e tests account for it explicitly.

18.7 The Capability Model

Tauri 2 is permission-based: the WebView can do nothing native unless a capability explicitly grants it. The grants live in capabilities/main.json, and they are deliberately minimal.

CapabilityWhy it is granted
core:defaultStandard window and event APIs.
shell:allow-spawnSpawn the rb-backend sidecar — and only that. There is deliberately no allow-execute, which would permit arbitrary command execution.
dialog (open / save)Native file and folder pickers.
fs (write / read / mkdir)The native filesystem writes used by the export modal.
▲
spawn, not execute

The distinction between shell:allow-spawn and shell:allow-execute is a security decision. spawn permits launching the one named sidecar; execute would permit running arbitrary commands — a remote-code-execution surface. The application grants only what it needs, and the comment in the JSON file says so.

18.8 Rust Coding Rules

The Rust layer is held to a strict rulebook, much of it the product of Phase 2 of the Slopcode-Cleanup:

RuleDetail
Every command returns Result<T, String>A command must never panic — a lock-poison or an error is mapped to a string the frontend can show.
No .unwrap() / .expect() in fallible pathsErrors propagate with ? or typed enums; expect() is allowed only in main() setup with a clear message.
No println! / eprintln!Logging goes through the log facade — log::info!, warn!, error! — never the print macros, outside tests.
Typed errors over Box<dyn Error>Error types are thiserror enums — for example the SoundCloud client's ScError.
No naked unsafe impl Send + SyncAny unsafe needs a // SAFETY: justification — and the audio engine was refactored specifically to remove one.
Every command documentedEvery #[tauri::command] carries a /// doc comment with an # Errors section.
cargo clippy -- -D warningsMust be clean — every warning is an error. No #[allow(...)] without a documented reason.

These rules are enforced by the rust-lint-test CI job, which runs clippy and the Rust tests on every push (Chapter 23).

Chapter summary

The Rust/Tauri layer is the application's host process: it creates the window, spawns and supervises the Python sidecar, captures the session token from the sidecar's stdout (and scrubs it from the logs), and exposes a tight set of IPC commands and events to the frontend. Its capability grants are minimal — spawn the one sidecar, never execute. A strict rulebook — typed results, no panics, no print macros, clippy-clean — keeps the native layer trustworthy.
