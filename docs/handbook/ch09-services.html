Chapter 09 · The Backend

Business Logic & Services

Between the thin route handlers and the database sits the business-logic layer — a set of focused service classes, an import pipeline, a smart-playlist evaluator, and the Rekordbox interoperability modules.

9.1 The Role of the Service Layer

A well-built FastAPI application keeps its route handlers thin. A handler validates input, calls into business logic, and shapes the response — it does not contain the business logic. In this backend that logic lives in a service layer, and the centrepiece is app/services.py.

main.py — thin"] R --> S["Service classes

folder_watcher · import_tracker"] R --> SP["Smart-playlist engine"] S --> DB["Database layer"] S --> FF["FFmpeg / filesystem"] S --> TAG["audio_tags — mutagen"]

9.2 The Service Classes

app/services.py defines ten business-logic classes. Each is instantiated once at import time and groups a coherent set of operations.

Two of these deserve a closer look because they embody coding rules from Chapter 7.

AudioEngine — every subprocess has a timeout

AudioEngine shells out to FFmpeg for conversion, render, and slicing. Every one of those calls carries an explicit timeout= and logs the command before it starts and the return code and elapsed time after — a direct outcome of the Slopcode-Cleanup, which found subprocess calls with no timeout at all. A hung encoder can no longer hang the backend.

ImportManager — honest metadata

ImportManager reads native tags via the audio_tags module first, and only if those are absent falls back to splitting the filename on Artist - Title. An earlier version hardcoded placeholder strings like "New Import" — the cleanup replaced that with the honest read-then-fall-back behaviour.

9.3 Reading and Writing Tags

app/audio_tags.py is the module that talks to the bytes inside audio files, built on the mutagen library. It is mode- and format-agnostic:

FunctionBehaviour
read_tags(path)Returns a dict — title, artist, album, genre, year, comment, BPM, key, ISRC — by probing format-specific keys, then falling back to filename parsing.
write_tags(path, updates, artwork)Mirrors metadata edits back into the file, handling ID3, FLAC, MP4, Vorbis, AIFF, and WAV containers.

This module is what keeps the library's metadata and the files on disk in agreement — when a user edits a track's genre in the UI, the change can be written through to the file's actual tags.

9.4 The Import Pipeline

Local-file import is more than a single function call — it is a pipeline with visible stages, built from two cooperating modules.

The folder watcher

app/folder_watcher.py implements FolderWatcher, which auto-imports audio from user-configured folders. For each watched folder it does two things:

  1. An initial recursive scan, which catches files added while the application was closed.
  2. A watchdog observer, which fires on file creation or a move into the folder.

New files are debounced — the watcher waits for filesystem events on a path to settle before importing, so a file still being written is never analysed half-complete. Imports are then handed to ImportManager on a small worker pool, and path-based deduplication stops repeated events from re-importing the same file.

The import tracker

app/import_tracker.py is a thread-safe singleton that gives the frontend a live, transparent view of the import pipeline. Every file moves through five tracked stages:

stateDiagram-v2 [*] --> Queued Queued --> Analyzing: worker picks it up Analyzing --> Importing: BPM / key / beats / cues done Importing --> ANLZ: in library, artwork attached ANLZ --> Completed: DAT / EXT / 2EX written Completed --> [*]

This is what powers the Import Manager view and the sticky progress banner from Chapter 3 — the user can watch every track move from Queued to Completed.

9.5 The Smart-Playlist Engine

app/smart_playlist_engine.py evaluates rule-based playlists. It is deliberately mode-agnostic — it works on any list of track dictionaries, so the same evaluator serves both XML and Live mode.

Its criteria format is a simplified, Rekordbox-XML-compatible structure:

{
  "LogicalOperator": "all" | "any",
  "AutomaticUpdate": "1" | "0",
  "conditions": [
    {
      "Field": "<field-id>",
      "Operator": "<op>",
      "ValueLeft": "...",
      "ValueRight": "..." | null,
      "ValueUnit": "..."
    }
  ]
}

The field IDs follow the Rekordbox specification subset — Title, Genre, Rating, BPM, key, date-added, and others. The LogicalOperator chooses whether a track must satisfy all conditions or any. A _coerce() helper normalises a value to the type a given field expects before comparison, so a numeric BPM rule and a string genre rule are both handled correctly.

◆
Why mode-agnostic matters

Because the engine operates on plain track dicts, a smart playlist defined in Standalone XML mode behaves identically if the user later switches to Live mode. The rule logic never needs to know where the tracks came from — a recurring theme of the whole backend.

9.6 Rekordbox Interoperability

A cluster of modules exists purely to speak Rekordbox's own formats — importing from and exporting to the files Rekordbox itself produces.

ModuleRole
rekordbox_export.pyConverts analysis-engine results into Rekordbox XML TRACK elements, with TEMPO nodes for the beatgrid and POSITION_MARK nodes for cues.
rekordbox_bridge.pyThe high-level bridge — export a selection of tracks to XML, or import tracks from a Rekordbox XML export.
xml_generator.pyThe RekordboxXML class — generates a valid DJ_PLAYLISTS XML document from Python track data, with dynamic and static beatgrids and cue points.
rbep_parser.pyParses and serialises .rbep project files — volume envelopes, BPM maps, hot cues, memory cues, beatgrids.
batch_worker.pyA command-line tool for batch comment/tag updates directly against master.db.

The .rbep project format

The .rbep file — Rekordbox Editor Project — is the non-destructive editor's overlay format. The parser is real-format aware: it understands that a <filepath> sits directly under <song>, that an <edit> sits directly under <track>, and that <position> data is a list, not a single object. It exposes the editor's timeline duration through positions[] and editEndBeats. Chapter 16 covers how the DAW produces and consumes these files.

9.7 The Sidecar Storage

Not all data belongs in the library database. app/sidecar.py provides SidecarStorage, which persists supplementary data — such as an artist's SoundCloud links and other custom fields — in an app_data.json sidecar file. It is a deliberately simple key-value store that keeps the main database clean of data that is not part of the Rekordbox schema.

9.8 Analysis Settings

app/analysis_settings.py centralises every tunable parameter of the analysis pipeline — BPM output ranges, frame sizes, thresholds. The defaults match Rekordbox conventions, and every value can be overridden with an environment variable prefixed RB_ANALYSIS_ — for example RB_ANALYSIS_BPM_OUTPUT_MIN=70. The settings are held in a frozen dataclass, so the analysis engine reads a consistent, validated configuration rather than scattering magic numbers through its code.

9.9 How the Layers Cooperate

A concrete example ties the chapter together — importing one local file:

sequenceDiagram participant H as Route handler participant IM as ImportManager participant TAG as audio_tags participant AN as Analysis engine participant IT as import_tracker participant DB as Database layer H->>IM: import_file(path) IM->>IT: register — stage Queued IM->>TAG: read_tags(path) TAG-->>IM: title, artist, genre, ... IM->>IT: stage Analyzing IM->>AN: run full analysis AN-->>IM: bpm, key, beatgrid, cues IM->>IT: stage Importing IM->>DB: add_track(track_data) IM->>IT: stage Completed IM-->>H: the new track

This is the pattern the whole backend follows: a thin handler, a service that orchestrates, focused modules that each do one thing well, and a tracker that keeps the user informed. The next chapter goes one layer deeper — into the database that add_track writes to.

Chapter summary

The service layer keeps route handlers thin. services.py holds ten business-logic classes; the import pipeline pairs a debounced folder watcher with a five-stage tracker; the smart-playlist engine evaluates Rekordbox-compatible rules on mode-agnostic track dicts; and a cluster of interop modules speaks Rekordbox's own XML and .rbep formats. Every subprocess carries a timeout, every import is transparent, and no logic leaks into the routes.
