Chapter 03 · Orientation

The Complete Feature Catalog

Every capability the application ships, organised into ten functional domains. This chapter is the exhaustive inventory; later chapters explain how each capability is built.

3.1 How the Catalog Is Organised

The application is large. Counting only user-facing capabilities — not the internal machinery — it ships well over a hundred distinct features. To keep the catalogue navigable, every feature is filed under one of ten domains:

flowchart TB APP(["Music Library Manager"]) APP --> D1["1 · Library Management"] APP --> D2["2 · Playlists & Smart Playlists"] APP --> D3["3 · SoundCloud Integration"] APP --> D4["4 · Local Import"] APP --> D5["5 · Audio Analysis"] APP --> D6["6 · Editing & the DAW"] APP --> D7["7 · USB Export"] APP --> D8["8 · Library Health & Tools"] APP --> D9["9 · Playback & Search"] APP --> D10["10 · Settings & System"]

Each domain section below lists its features in a table, and the most significant domains are also illustrated with a diagram. Where a feature maps to a later chapter, the table says so.

3.2 Domain 1 — Library Management

The library is the heart of the application: the curated, analysed collection of tracks. It can be managed in either of two modes, and a single wrapper hides the difference from every other feature.

FeatureDescription
Standalone XML modeThe application's own Rekordbox-compatible XML library. No Rekordbox install required.
Live Rekordbox modeDirect read/write of an existing Rekordbox master.db via the pyrekordbox library.
Four-card mode pickerOn startup the user chooses: New Empty library, Standalone, Import an existing XML, or a Defined Path.
Mode-agnostic CRUDA wrapper routes every create/read/update/delete to the active mode — no feature contains mode branching.
In-memory cacheThe XML library is parsed once into a dict cache; subsequent reads are sub-five-millisecond.
Library lifecycleLoad, unload, create-new, and sync (persist) operations on the whole library.
Track metadata editingTitle, artist, album, genre, year, comment, rating, colour, BPM, key — edited inline or in a metadata panel.
Grouping browsersBrowse the collection by artist, genre, label, or album, each with its own track listing.
Artwork servingEmbedded cover art is extracted and served to the UI for every track.
Inline ratingClick-to-rate with a five-star control directly in the track table.
Pioneer colour tagsA nine-colour tag picker matching the CDJ colour palette.

The library layer is documented in full in Chapter 10.

3.3 Domain 2 — Playlists & Smart Playlists

Playlists are how a DJ slices the library into sets. The application supports both ordinary playlists and rule-driven smart playlists, nested in a folder tree.

FeatureDescription
Playlist CRUDCreate, rename, delete, and duplicate playlists.
Folder treePlaylists nest inside folders; the whole structure is a tree.
Drag reorderPlaylists and the tracks inside them can be dragged into a new order.
Move within treeA "move to" picker relocates a playlist or folder anywhere in the tree.
Track membershipAdd and remove tracks; reorder tracks within a playlist.
Smart playlistsRule-based playlists with field/operator/value/unit conditions and an AND/OR combinator.
Smart-playlist editorA modal with field rows, a live preview, and Rekordbox-XML-compatible criteria.
Smart-playlist re-evaluationSmart playlists re-evaluate against the current library on demand.
Context menusRight-click menus — seven actions on playlists, nine on tracks — built to work inside the Tauri WebView.
◆
Smart-playlist criteria

The smart-playlist engine accepts a Rekordbox-XML-compatible criteria object — a logical operator (all/any) plus a list of conditions, each naming a field (Title, Genre, Rating, BPM, key, date-added and more), an operator, and one or two values. Chapter 9 covers the evaluator.

3.5 Domain 4 — Local Import

The second acquisition path takes files already on disk. Its defining characteristic is transparency: the user can watch every track move through a seven-stage pipeline.

FeatureDescription
Drag-and-drop importDrop audio files or folders directly onto the import view.
Folder-picker importBrowse the filesystem and select a folder to import.
Folder-as-playlistThe imported folder's name becomes the playlist; every file — duplicates included — is bundled together.
Folder watcherWatched folders auto-import new files: an initial recursive scan plus a live filesystem observer.
Debounced importNew files wait until writing settles before import, so half-written files are never analysed.
Bulk path importImport an explicit list of file paths in one call.
Folder scanA background scan of a directory that auto-imports any audio not yet in the library.
Import trackerA thread-safe tracker exposing each file's live stage to the UI.
Import Manager viewA dedicated view showing every track's stage with per-stage timestamps.
Sticky progress bannerA bottom-of-screen banner, visible on every view, showing aggregate import progress.

3.6 Domain 5 — Audio Analysis

Analysis is what turns a plain audio file into a DJ-usable track. The engine extracts tempo, musical key, the beatgrid, phrase structure, automatic cue points, and multi-resolution waveforms.

FeatureDescription
BPM detectionTempo estimation via a madmom recurrent-neural-network beat tracker, with a librosa fallback.
Key detectionMusical key via the essentia key extractor, with a Krumhansl-Schmuckler / Temperley ensemble fallback.
BeatgridA full beatgrid of beat positions and per-segment tempo.
Phrase detectionPhrase and bar markers derived from the beatgrid.
Auto hot-cuesHot cues A–H placed automatically at musically meaningful positions.
Auto memory-cuesMemory cues placed automatically from the analysis.
Waveform generationSeven waveform variants — mono previews, colour previews, and 3-band detail data.
Analysis cacheA persistent gzipped cache keyed on file mtime, size, and analyzer version — unchanged files are never re-analysed.
Single-track analysisAnalyse one track on demand, with a polling task ID.
Analyze-fullOne call that analyses, writes ANLZ files, and updates master.db.
Batch analysisAnalyse many tracks with streaming NDJSON progress.
Quality presetsFast (librosa), Standard (madmom), or Thorough (ensemble) analysis quality.
Tunable settingsAnalysis parameters overridable via RB_ANALYSIS_ environment variables.

The analysis engine is the subject of Chapter 11; the ANLZ files it feeds are Chapter 12.

3.7 Domain 6 — Editing & the DAW

The application includes a genuine non-destructive editing workstation — a four-panel DAW — plus a standalone waveform editor. Crucially, no edit ever rewrites the source audio file.

FeatureDescription
Four-panel DAWThe DJ Edit DAW: toolbar, timeline, browser, and control strip.
Non-destructive editorA region-based editor for cuts, inserts, deletes, and pastes.
Waveform editorA WaveSurfer-based editor with sample-accurate, FFmpeg-backed seeking.
Hot cuesEight colour-coded hot cues (A–H), placeable and jumpable.
Memory cuesTime-sorted memory cues.
LoopsLoops with active state and colour.
Beatgrid editingInteractive beatgrid adjustment on the timeline.
Volume envelopesDraggable fade-in, fade-out, and gain nodes per region.
Undo / redoFull-state-snapshot history.
Snap to gridEdits snap to beat divisions.
Project filesEdits persist to .rbep overlay project files — the source audio is untouched.
ExportRender the edited timeline to WAV, MP3, or FLAC with optional normalisation.
Configurable shortcutsFourteen keyboard actions, each rebindable in settings.
3-band waveformLow/mid/high frequency-band rendering, CDJ-style.

The DAW is Chapter 16; the waveform system is Chapter 17.

3.12 Feature Maturity

Not every feature is at the same stage. The catalogue above is shipped and working; a parallel set of capabilities is in active research. A bird's-eye view:

The twelve pipeline features — recommenders, a mobile companion, external track matching, metadata repair — are catalogued with full technical detail in Chapter 25.

Chapter summary

The application ships well over a hundred user-facing features across ten domains: library management, playlists, SoundCloud, local import, audio analysis, editing and the DAW, USB export, library health, playback and search, and settings and system. The USB-export and analysis domains are the most technically demanding and earn their own deep-dive chapters later. Twelve further features are in the research pipeline.
