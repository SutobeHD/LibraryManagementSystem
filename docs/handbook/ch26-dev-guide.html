Developer Guide

Everything a new contributor needs — setting up the three stacks, the navigation docs, the coding rules, the commit conventions, and the workflow for adding a route or a component.

26.1 Setting Up

A working development environment needs the three language toolchains plus FFmpeg on the PATH.

PrerequisiteFor
Python 3.10+The backend and its analysis stack.
Node.jsThe frontend (Vite, React).
The Rust toolchainThe Tauri shell and the native audio engine.
FFmpegAudio conversion — must be on the system PATH.

The dependency installs follow the deterministic, supply-chain-safe commands of Chapter 21 — npm ci, never npm install; pip install -r requirements.txt; cargo build from the committed Cargo.lock.

▲
First, set the git identity

In a fresh clone, verify the per-repository git identity before the first commit. The project requires the GitHub no-reply email — the name SutobeHD and the matching @users.noreply.github.com address. GitHub's "block command-line pushes that expose my email" setting rejects pushes otherwise. Set it per-repo, never globally.

26.3 Finding Your Way Around

The repository ships a graduated set of navigation documents — start cheap, go deeper only as needed.

flowchart TB L1["docs/MAP.md — L1
every file, one-line purpose"] L1 --> L2["docs/MAP_L2.md — L2
L1 plus public classes & functions"] L2 --> L3["docs/backend-index.md · frontend-index.md · rust-index.md — L3
routes, components, commands in detail"] L3 --> FM["docs/FILE_MAP.md
curated map with per-file invariants"] FM --> ARCH["docs/architecture.md
system diagrams & data flows"]

The L1 and L2 maps are regenerated deterministically from the source by scripts/regen_maps.py; CI fails if they drift. The FILE_MAP.md is hand-curated and carries the narrative invariants — it is the place to learn why a file is the way it is before changing it. This handbook is the most complete of all these documents and complements rather than replaces them.

26.4 The Coding Rules

The project's rules live in .claude/rules/ and are non-negotiable — they are load-bearing constraints, not style preferences. The essentials, by stack:

StackCore rules
PythonPydantic v2 (.model_dump()); no bare except; no requests in async; every subprocess has a timeout; no raw f-string SQL; type hints on new code; pathlib over os.path.
RustNo .unwrap() in fallible paths; no println!; typed thiserror errors; no naked unsafe impl Send + Sync; every command documented; clippy-clean.
FrontendNo alert/confirm/prompt; no raw fetch(); no stray console.log; no empty catch {}; no magic numbers.
AllExact-pinned dependencies; no --no-verify; comments explain why, never what.

26.5 The Self-Correction Loop

After every code edit, before the work is called done, the project's rules prescribe a five-step loop (introduced in Chapter 23): lint and format the changed file; run the tests that cover it; verify any UI change in a real browser; sync the navigation docs; summarise. A step that fails is fixed at the root — never bypassed.

26.6 Adding a Route

Adding a FastAPI route is a guided process, because a route must apply several conventions at once — authentication, the write lock, a Pydantic model, typed errors, path validation.

flowchart LR P["Plan the route
route-architect agent"] P --> M["Define a Pydantic v2 model"] M --> A["Add Depends(require_session)
for any mutation"] A --> L["Acquire _db_write_lock
for any master.db write"] L --> V["validate_audio_path
for any client path"] V --> D["Update backend-index.md + tests"]

The /route-add slash-command and the route-architect subagent exist precisely to apply this checklist automatically — they produce ready-to-paste route code with the conventions already in place, plus a list of the downstream updates (the backend index, the file map, the tests, the frontend client).

26.7 Adding a Component

A new React component must: handle its loading and error states explicitly (the mandatory contract of Chapter 15); fetch only through the shared axios client; use the shared logger, not bare console; and, if it is a view, be registered in the lazy-load tab router in main.jsx. A UI change is not done until the e2e-tester agent has driven it in a real browser — a type-check passing is not the same as a feature working.

26.8 Committing

The project commits intensely and atomically — one logical change per commit, committed as soon as the tree is green. Messages follow Conventional Commits:

<type>(<scope>): <imperative summary, under 70 chars>

<optional body — the why, not the what>
ElementValues
Typesfeat, fix, refactor, docs, test, chore, perf, build, ci, revert
Scopesbackend, frontend, tauri, audio, usb, pdb, anlz, analysis, db, docs, deps

History is kept revertable: every commit lands as a normal commit, every push is a fast-forward, and a mistake is undone with git revert, not a history rewrite. Force-pushing, amending a pushed commit, and --no-verify are forbidden.

26.9 The Research-First Rule

Not every change goes straight to code. The rule: a feature that touches two or more modules, or has more than one plausible approach, starts as a research document in the pipeline of Chapter 24 — not in an editor. One-off bug fixes, single-file refactors, plain questions, and doc edits skip the pipeline. When in doubt, check docs/research/_INDEX.md first: there may already be an in-flight document for the area, and contradicting it is not allowed.

26.10 The Autonomy Model

The repository is configured for highly autonomous agentic development. The boundary is drawn by reversibility: local, reversible actions — editing files, running tests, building, committing — are done freely; actions that are destructive, irreversible, or change shared state — force-pushing, dropping data, deleting branches, creating PRs, installing new dependencies — require explicit confirmation. The .claude/settings.json allowlist encodes exactly this split.

◆
Streamlining bias

The repository culture favours automating recurring manual rituals. Spotted a multi-step bookkeeping chore? The expectation is to automate it — a slash-command, a hook, a routine — and show the change, rather than keep doing it by hand. The slash-commands and the pipeline routines of Chapter 24 are the accumulated result of this bias.

Chapter summary

A contributor needs the three toolchains plus FFmpeg, the correct per-repo git identity, and the navigation docs (the L1/L2 maps, the index docs, the file map, this handbook). The coding rules are load-bearing and stack-specific; a five-step self-correction loop runs after every edit. Routes and components follow guided checklists; commits are atomic and Conventional; history stays revertable. Multi-module features start in the research pipeline, and the autonomy model is drawn along the line of reversibility.
