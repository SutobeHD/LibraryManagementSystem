Chapter 07 · The Backend

Backend Overview (FastAPI)

The backend is a Python FastAPI application — roughly forty modules and 4,000 lines of route definitions — that runs as a loopback-only sidecar. This chapter establishes the shape, the conventions, and the request lifecycle before the deep dives that follow.

7.1 What the Backend Is

The backend is the application's centre of gravity. It owns the library, the database connections, the analysis pipeline, the USB writers, the SoundCloud integration, and every byte of disk I/O. The frontend has no direct access to the filesystem or the database — it asks the backend, over HTTP, for everything.

Physically, the backend is the app/ Python package: about forty modules, with app/main.py as the entry point that wires them all into a single FastAPI application. In development it runs as python -m app.main; in production it is a PyInstaller-built executable named rb-backend. Either way it serves 127.0.0.1:8000 and nothing else.

7.3 The Request Lifecycle

Every request that reaches the backend travels the same path. Understanding it once explains the behaviour of all 146 routes.

sequenceDiagram participant C as Client (frontend) participant CORS as CORS Middleware participant RL as Rate Limiter participant AUTH as require_session participant H as Route Handler participant E as Error Handler C->>CORS: HTTP request CORS->>CORS: origin allowed? else reject CORS->>RL: (HIGH-tier routes only) RL->>RL: token-bucket check, else 429 RL->>AUTH: (mutation routes only) AUTH->>AUTH: Bearer token valid? else 401 AUTH->>H: dispatch to handler H->>H: Pydantic validates the body alt success H-->>C: { status: "ok", data: ... } else exception raised H->>E: exception bubbles up E-->>C: { status: "error", message, code } end

Three of those stages — CORS, the rate limiter, and the auth gate — are security machinery and are fully documented in Chapter 21. The remaining two — validation and the response envelope — are conventions worth establishing now.

7.4 Typed Requests with Pydantic v2

Every endpoint that accepts a request body declares a Pydantic v2 model for it. This was not always true — the Slopcode-Cleanup found sixteen endpoints accepting an untyped dict — and it is now a hard rule. A typed model gives FastAPI automatic validation, automatic 422 responses on bad input, and a single place to document the shape.

The settings endpoint is the most carefully modelled. Its SetReq model does not merely validate types; it enforces strict size caps — 8 KB per value, 64 dictionary keys, 256 list items, 256 KB total — because the endpoint writes to a persistent store and an unbounded write would be a denial-of-service primitive (Chapter 21).

7.5 The Response Envelope

Every endpoint returns one of two shapes, so the frontend can handle success and failure uniformly:

# Success
{ "status": "ok", "data": ... }

# Error
{ "status": "error", "message": "Human-readable description", "code": "ERROR_CODE" }

The code field is a stable, machine-readable error identifier. The defined codes:

CodeMeaning
REKORDBOX_RUNNINGA write was refused because Rekordbox has the database open.
PATH_SANDBOX_VIOLATIONA file path fell outside the allowed roots.
AUTH_EXPIREDThe SoundCloud OAuth token has expired.
RATE_LIMITEDThe SoundCloud API returned 429.
FILE_NOT_FOUNDAn audio file is missing from disk.
ANALYSIS_PENDINGAn analysis task is still running.
ANALYSIS_FAILEDAn analysis task failed.
USB_LOCKEDAnother USB sync is already in progress.

7.6 The Filesystem Sandbox

The single most important safety function in the backend is validate_audio_path(). Many endpoints take a file path from the client — to stream audio, to import a file, to reveal a location in the OS file manager. An unchecked path is a path-traversal vulnerability.

validate_audio_path() resolves the requested path and checks it against ALLOWED_AUDIO_ROOTS using Path.is_relative_to(resolved_root) — not a string startswith, which is defeatable with .. tricks and symlinks. A path outside the sandbox raises a ValueError and the request is refused with PATH_SANDBOX_VIOLATION.

▲
The sandbox is a coding rule, not an option

The project's rules forbid bypassing the sandbox with os.path.normpath tricks. Any new endpoint that touches a client-supplied path must route it through validate_audio_path(). The 2026-05-18 security work fixed a /api/file/reveal endpoint that had skipped this — see Chapter 21.

7.7 The Database Write Lock

FastAPI serves requests on a thread pool, so two requests can run at once. The Rekordbox master.db cannot tolerate that — the underlying library crashes on concurrent writes. The backend's answer is _db_write_lock, a module-level re-entrant lock in app/main.py.

Every method that writes the Rekordbox database acquires this lock. The database module even applies it automatically: at import time a decorator pass wraps every mutating method of the RekordboxDB class so the lock is taken without the caller having to remember. Routes that need to perform several mutations atomically use an exposed db_lock() context manager. The lock is re-entrant, so a wrapped method calling another wrapped method does not deadlock. Chapter 10 covers the full concurrency story, including a known coverage gap that is on the roadmap.

7.8 Error-Handling Conventions

The Slopcode-Cleanup left the backend with a strict set of error-handling rules, all of which are still enforced:

RuleRationale
No bare except: or except: passEvery catch names its exception type and logs it. The audit found 40+ bare excepts; all were typed.
No synchronous requests.get() inside async defA blocking call freezes the event loop. Async paths use an async HTTP client with a timeout.
Every subprocess call has a timeout=A hung FFmpeg or PowerShell process must never hang the backend. FFmpeg defaults to 30 s.
No raw f-string SQLSQL is parameterised or goes through the ORM. Where concatenation is unavoidable, an explicit table allowlist guards it.
Type hints required on new codemypy must pass before commit.
pathlib over os.path.joinConsistent, safe path handling throughout.

Logging follows from the same era: structured log lines with key=value context, and a RedactingFormatter that strips absolute filesystem paths from every log line and traceback before it is written — so a crash report or a shared log file cannot leak a user's directory structure.

Analysis parameters have their own configuration module, analysis_settings.py, whose defaults match Rekordbox conventions and can be overridden with RB_ANALYSIS_-prefixed environment variables. Secrets — chiefly the SoundCloud client credentials — come from a .env file that is never committed; .env.example is the only template in the repository.

7.10 The Startup Sequence

When the backend process starts, before it serves a single request, it:

  1. Generates the session token and emits the LMS_TOKEN= banner (Chapter 21).
  2. Initialises the download registry database schema.
  3. Configures CORS for the allowed localhost origins.
  4. Installs the global exception handlers.
  5. Binds Uvicorn to 127.0.0.1:8000.

The library itself is not loaded at startup — it is loaded on demand when the user picks a mode, so the backend is responsive immediately and the heavy parse happens only when needed.

Chapter summary

The backend is a ~40-module FastAPI application with app/main.py as the hub for 146 routes. Every request passes CORS, optionally the rate limiter, optionally the auth gate, then the handler — with a global error handler producing a uniform {status, data} or {status, message, code} envelope. Three pieces of machinery underpin everything: the validate_audio_path() sandbox, the _db_write_lock, and a strict set of error-handling conventions from the cleanup era.
