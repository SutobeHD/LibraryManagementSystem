Testing & Quality Assurance

Over 200 Python tests, byte-fidelity guards on the Pioneer formats, Rust and frontend test suites, two end-to-end channels, and a four-language linting gate — the safety net under every change.

23.1 The Testing Philosophy

Before the Slopcode-Cleanup, the project had three tests for fifty modules. Phase 6 of that mission set out to fix it, and the result is a layered safety net. The philosophy is straightforward: anything that is hard to get right and catastrophic to get wrong gets a test that fails loudly. The byte-exact Pioneer formats and the security gates are tested most heavily, precisely because their failure modes are silent.

The shared auth fixture

tests/conftest.py carries the suite's most important piece of shared machinery: an autouse auth_token fixture. Because every mutation route now requires a Bearer token, the fixture monkeypatches the session token to a known constant and hands tests a ready-made Authorization header — so a gated-route test stays short. Tests that need to exercise the genuine 401 path opt out with a @pytest.mark.no_auth marker.

23.3 Byte-Fidelity Tests

Two test files are unlike the rest. They do not check behaviour — they check bytes.

◈
test_pdb_structure.py — the format guardian

This file asserts every structural invariant of the PDB writer of Chapter 13 — page flags, the per-table blank pages, the chain terminator, the free_size arithmetic. The project's troubleshooting rule is blunt: if this test fails, the change is wrong — not the test. The reference fixture is a real Pioneer drive's bytes, and it is not updated without understanding exactly why.

test_onelibrary_wal_flush.py is the companion: it verifies that after an OneLibrary write the SQLite write-ahead log is zero bytes — catching the WAL-flush bug Chapter 13 described. These two tests are why the USB export can be trusted between releases.

That purity (Chapter 17) makes it trivially testable, and the test file exercises the region, transport, selection, cue, and history transitions.

23.5 End-to-End Testing — Two Channels

Unit tests prove a function is correct. They do not prove a feature works — that a click does the right thing in a real browser. For that the project has two end-to-end channels, and a feature change picks the right one.

flowchart TB Q{"Needs a native dialog,
OAuth callback, or
the audio engine?"} Q -->|"no"| A["Channel A — Web Preview
Vite + FastAPI in a headless browser
fast, no build step"] Q -->|"yes"| B["Channel B — Tauri WebDriver
the real WebView2 window
full native APIs, needs a build"]
ChannelDrivesBest for
A — Web PreviewThe Vite dev server in a headless browser.UI, state, axios calls, FastAPI integration — the default, no build needed.
B — Tauri WebDriverThe real Tauri WebView2 window via tauri-driver and msedgedriver.Native dialogs, OAuth, the filesystem, the audio engine.

Channel A is cheap and fast because it skips the build entirely; its limitation is that window.__TAURI__ is undefined, so anything calling invoke() takes a fallback path. Channel B is the real thing — the bundled application driven by Selenium — but it requires a build and the right WebView2 driver version. The tests/e2e/ directory holds a Mocha + Selenium smoke test that boots the built application and confirms the mode picker renders.

23.7 The Self-Correction Loop

Testing is not only a CI gate — it is a development discipline. The project's coding rules define a self-correction loop that runs after every code edit, before the work is called done:

flowchart LR E["Edit code"] --> L["1 · Lint & format
the changed file"] L --> T["2 · Run the tests
covering the change"] T --> U["3 · Verify UI changes
in a browser"] U --> D["4 · Sync the docs"] D --> S["5 · Summarise"] T -.->|"failure"| FIX["Fix the root cause"] FIX --> L

The loop is supported by specialised review agents — a test-runner that parses and classifies failures, an e2e-tester that drives the real UI, an audio-stack reviewer that runs the full audio toolchain. They are the subject of Chapter 24's wider story about how the project develops itself.

Chapter summary

Quality assurance is four layers: a 200-plus-test pytest suite with a shared auth fixture, two byte-fidelity tests that guard the Pioneer formats against silent corruption, Rust and frontend unit tests (the pure DAW reducer is the natural target), and two end-to-end channels — fast Web Preview and full Tauri WebDriver. Four static toolchains gate every push, and a post-edit self-correction loop turns testing into a development habit rather than an afterthought.
