Chapter 24 · Quality & Operations

The R&D Pipeline

The project develops itself. A multi-agent research pipeline — five autonomous routines, four user sign-off gates — advances every feature from raw idea to merged code, and it is itself part of the codebase.

24.1 An Unusual Chapter

Most handbooks document a product. This chapter documents a process — but a process so thoroughly encoded in the repository, in versioned files and slash-commands and agent definitions, that it is as much a part of the system as any module. The pipeline is how Chapter 25's roadmap gets built.

The idea is simple to state. Every feature is a Markdown document. That document moves through a fixed sequence of states. Autonomous routines do the work of advancing it; the user keeps a small number of deliberate sign-off gates. The state is encoded in the document's folder and filename prefix — the file system is the single source of truth.

24.2 The State Chain

A feature document lives in one of three folders — research/, implement/, archived/ — and carries a prefix that names its state.

parallel research agents"] E --> GB{"midgate_ — GATE B"} GB --> V["evaluated_

implementation plan + task queue"] DP --> R["review_
plan-verification agent"] R --> GC{"plangate_ — GATE C"} GC --> AC["accepted_
task queue active"] AC --> IP["inprogress_
routine writes code on branches"] IP --> GD{"open PR — GATE D"} GD --> IM["archived/implemented_
shipped"]

There are two kinds of state. A work state is one a routine actively works — and the routine itself performs the git mv to the next state when it is done. A gate state (the *gate_ prefixes) is a pause: the work stops and waits for the user.

24.3 The Four Gates

Between the autonomous stretches sit four deliberate human checkpoints. Only the user advances a gate.

GateStateThe user confirms
Aideagate_The worked-up idea matches intent, and the research plan is right.
Bmidgate_The research is heading the right direction (a mid-research checkpoint).
Cplangate_The implementation plan and its task queue are approved.
Dan open PRThe branch is reviewed and CI is green — order the merge to main.

Gates A, B, and C are advanced with slash-commands — /gate-pass or /gate-reject. Gate D is a normal pull-request review. The iron rule: a routine never advances a gate, and never merges or rebases to main.

24.4 The Five Routines

The work between gates is done by five routines — scheduled claude.ai/code sessions, each triggered by a cron timer, each reading one work state and spawning its own sub-agents.

24.5 Routines Write Code — Within Bounds

One routine, research-implement, writes actual application code. This is powerful, so it is tightly bounded:

  • Only on a document in the inprogress_ state.
  • Only on routine/<slug>-task-<N> branches — never main.
  • Only on task-queue items that were approved at GATE C — no freelancing.
  • One task is one small PR; CI and a review agent gate it; the user merges at GATE D.

The other routines — draft, explore, plan, triage — touch only the research documents. No routine modifies app/, frontend/, or src-tauri/ outside an approved task queue.

24.6 Multi-Agent Mechanics

A routine is a Claude Code session with the ability to spawn sub-agents. "Multiple agents" means the routine fans work out to focused, parallel sub-agents and then collects the results.

flowchart TB R["research-explore routine"] R --> A1["Research agent · Open Question 1"] R --> A2["Research agent · Open Question 2"] R --> A3["Research agent · Open Question 3"] A1 --> C["Findings collected into the doc"] A2 --> C A3 --> C C --> VER["Verification agent
checks findings vs the original idea"]

A principle threads through every stage: idea fidelity. The ## Original Idea section, written once by the user, is never edited — and every verification agent checks the work against it, so the pipeline cannot drift away from what the user actually asked for.

24.7 The Development Subagents

Beyond the pipeline routines, the repository defines a set of specialised development subagents — used during ordinary work, not only by the routines. They are part of the project's "subagents default on" culture.

SubagentTriggered when

route-architectBefore adding or changing a FastAPI route — applies the auth, lock, and model conventions.

test-runnerBefore any commit — runs the tests and classifies failures.

24.9 Why This Matters to the Handbook

This chapter is not a digression. The twelve features in Chapter 25's roadmap are not a wishlist — they are twelve live documents currently moving through the pipeline described here. Ten are in the exploring_ state, deep in parallel research; two are at the idea_ stage. The seven security features of Chapter 21 went through an earlier form of the same process and are now archived as implemented_. The pipeline is the engine; the roadmap is what it is currently producing.

The project develops itself through a multi-agent research pipeline. Five cron-scheduled routines do the work — drafting, exploring, planning, implementing, triaging — each spawning parallel sub-agents and gated by verifiers.
