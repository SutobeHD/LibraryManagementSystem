Chapter 01 · Orientation

Introduction & Vision

Music Library Manager is a standalone desktop application that gives DJs full, local ownership of their music library — and the ability to export USB sticks that Pioneer CDJ hardware reads natively, without paying rent to anybody's cloud.

1.1 What This Product Is

Music Library Manager is a desktop DJ-library manager. It organises a DJ's collection of audio files — tagging, rating, grouping into playlists, analysing tempo and musical key, drawing waveforms, placing cue points — and then writes that collection out to a USB stick in the exact binary format that Pioneer's industry-standard CDJ-3000 players (and other modern CDJ/XDJ hardware) expect.

In the language of the DJ industry, it is a direct competitor to Rekordbox (Pioneer's own library software) and Serato DJ. The difference is one of philosophy. Those products are commercial, increasingly subscription-driven, and they hold a DJ's metadata inside a proprietary database whose format the vendor controls. Music Library Manager is built on the opposite premise: the library is yours, it lives on your disk, it works offline, and the format used to talk to the hardware is reverse-engineered, documented, and open.

A local-first, no-subscription desktop application that imports music from local folders or SoundCloud, analyses it to professional DJ standards, and exports CDJ-ready USB sticks — a fully open alternative to Rekordbox and Serato.

1.2 The Problem It Solves

To understand why this application exists, you need to understand the workflow of a working DJ and the friction baked into the incumbent tools.

The DJ's pipeline

A DJ does not simply "have music". A DJ maintains a library — a curated, analysed, annotated collection — and continuously moves slices of it onto the hardware they will actually perform with. That pipeline has four stages:

Where the incumbents create friction

The commercial tools handle this pipeline, but they impose costs that have nothing to do with the music:

The core insight of the project is that the genuinely hard part — the reverse-engineered binary format that a CDJ-3000 reads off a USB stick — is a solved problem. The Pioneer database (export.pdb) and the per-track analysis files (ANLZ0000.DAT and friends) have been studied by the open-source community for years. Once you can write those bytes correctly, there is no technical reason a DJ should have to pay a subscription to put music on a stick.

1.3 The Vision

The project is guided by four principles. They are not marketing slogans; they are constraints that shape every architectural decision in the chapters that follow.

①
Local-first
All data lives on the user's disk. The application functions completely with the network cable unplugged.
②
No subscription
No account, no licence server, no tiered features. The software is installed once and owned outright.
③
Open formats
The library can be exported to plain XML; the Pioneer binary formats are documented byte-for-byte.
④
Hardware-native
Output is verified against real Pioneer-exported sticks so a CDJ-3000 reads it without complaint.

"Local-first" deserves emphasis because it is unusual in 2026. Most software of this complexity assumes a backend in the cloud. Music Library Manager has a backend — a Python web server — but that server runs on the user's own machine, bound to the loopback interface, started and stopped by the desktop application itself. There is no remote component. The "cloud" is a folder on a local disk. This single decision drives the entire architecture, the security model, and the build pipeline.

The deliberate symmetry of these two workflows is an architectural statement: a track's origin should not change how it is treated once it is in the library. Whether a file arrived from SoundCloud or from a folder on disk, it goes through the same analysis engine, gets the same ANLZ files, and exports through the same USB writer. 1.6 Two Library Modes

The application can manage a DJ's collection in either of two modes, and a single abstraction layer hides the difference from every feature above it.

ModeBacking storeWhen to use it
Standalone XMLThe application's own Rekordbox-compatible XML file, parsed into an in-memory cache.The default. No Rekordbox install needed — the application is fully self-sufficient.
LiveAn existing Rekordbox master.db SQLite database, read and written through the pyrekordbox library.For DJs who already run Rekordbox and want this app to operate on the same collection.

A wrapper object routes every create/read/update/delete operation to whichever mode is active, so no feature in the codebase contains an if xml_mode … else live_mode … branch. The library layer is the subject of Chapter 10.

1.7 Design Philosophy

Beyond the four product principles, the codebase itself is shaped by a set of engineering convictions that recur throughout this handbook:

  • Right tool, right layer. Real-time audio is Rust. Offline DSP and orchestration are Python. The user interface is React. The boundaries are deliberate and Chapter 4 explains why each layer earns its place.
  • Non-destructive by default. Editing cue points, beatgrids, or volume envelopes never rewrites the source audio file. Edits live in overlay project files. The original recording is sacred.
  • Byte-exact interoperability. The Pioneer USB formats are not approximated. They are verified byte-for-byte against a real Pioneer-exported drive, because a single wrong flag silently corrupts a stick (Chapters 12 and 13).
  • Defensive isolation. A third-party library with a known crashing bug is quarantined in a subprocess so it can never take down the whole backend (Chapter 10).
  • Security as a first-class concern. Even though the server is loopback-only, every mutating endpoint is authenticated, every file path is sandboxed, and the supply chain is hardened (Chapter 21).

Chapter summary

Music Library Manager is a local-first, subscription-free desktop DJ tool that competes with Rekordbox and Serato by owning the curate → analyse → organise → export pipeline and writing CDJ-native USB sticks. It is built on four principles — local-first, no subscription, open formats, hardware-native — and serves club DJs, crate diggers, and library curators through two symmetric workflows and two interchangeable library modes.
