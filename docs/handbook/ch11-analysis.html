Chapter 11 · The Backend

The Audio Analysis Engine

Analysis is what turns a plain audio file into a DJ-usable track. The engine extracts tempo, musical key, the beatgrid, phrase structure, automatic cues, and seven waveform representations — all in background worker processes.

11.1 What Analysis Produces

When the analysis engine finishes with a track, it has produced everything a CDJ needs to display and mix that track intelligently:

BPM
Tempo, beats per minute
Key
Musical key, in Camelot notation
Grid
Beat positions + per-segment tempo
Cues
Phrase markers + auto hot-cues
Wave
Seven waveform representations
LUFS
Integrated loudness

This output then flows two ways: into the library database, and into the ANLZ binary files (Chapter 12) that get written to a USB stick.

11.2 The Engine and Its Entry Points

The engine is app/analysis_engine.py, exposing the AnalysisEngine class plus a top-level run_full_analysis() function. A second module, app/audio_analyzer.py, wraps it for HTTP task tracking.

Entry pointBehaviour
AnalysisEngine.submit(task_id, path)Queue a track for analysis in the background pool; returns immediately.
AnalysisEngine.get_status(task_id)Poll a task — returns status plus BPM, key, confidence, beatgrid when done.
AnalysisEngine.analyze_sync(path)Synchronous analysis — blocks; used only for scripting.
run_full_analysis(path)The top-level function — one call returns everything needed to write ANLZ files.

Every heavy call runs inside a ProcessPoolExecutor. DSP is CPU-bound, and Python's global interpreter lock means a thread pool would not actually parallelise it — separate processes do. This is also why a single analysis takes 2–8 seconds but the API stays responsive throughout.

11.4 Tempo — Beat Tracking

BPM detection is a two-tier affair. The primary tracker is madmom, which uses a recurrent neural network trained to find beats — markedly more accurate than spectral-flux methods, especially on electronic music with a strong but syncopated kick. If madmom is unavailable at import time, the engine falls back to librosa's beat.beat_track().

The output is not a single number. It is a beatgrid — a list of beat positions in time, plus the tempo of each segment, so a track that drifts or changes tempo is represented honestly rather than flattened to one average BPM.

11.5 Key — Harmonic Analysis

Musical key detection is likewise two-tier. The primary detector is essentia's key extractor; the fallback is an ensemble of the Krumhansl-Schmuckler and Temperley pitch-class-profile algorithms.

The key is reported in Camelot notation — the wheel-based system DJs use for harmonic mixing, where, for example, 8B is C major and 8A is A minor. Tracks whose Camelot codes are adjacent on the wheel mix together harmonically, which is why the UI colour-codes the key column by Camelot position.

◆
Why Camelot?

The Camelot wheel maps the 24 musical keys onto a clock face so that "what mixes with what" becomes a matter of stepping around the dial. Representing key this way makes the harmonic-mixing recommender on the roadmap (Chapter 25) a straightforward distance calculation.

11.6 Waveforms — Seven Representations

A waveform is not one thing. A CDJ shows an overview waveform and a zoomed detail waveform, in mono and in colour, and the analysis engine produces all of them. The 3-band colour comes from a Butterworth crossover — a filter that splits the signal into low, mid, and high bands at 200 Hz and 2500 Hz.

VariantWhat it is
PWAVA 400-point mono preview waveform.
PWV2A 100-point tiny preview.
PWV3A high-resolution mono detail waveform (150 frames per second).
PWV4A 1200×6 colour preview waveform.
PWV5A 16-bit colour detail waveform.
PWV6A 1200-point 3-band (low/mid/high) colour preview.
PWV7An N-point 3-band detail waveform (150 frames per second).

These names are not arbitrary — they are the tag identifiers of the Rekordbox ANLZ binary format. The analysis engine produces the data; the ANLZ writer of Chapter 12 packs it into the exact bytes Rekordbox expects.

11.7 Phrases and Automatic Cues

app/phrase_generator.py turns a beatgrid into musical structure. Dance music is built in phrases — typically 8, 16, or 32 bars — and marking those boundaries lets a DJ blend cleanly.

11.8 From Analysis to Database — the Orchestrator

app/analysis_db_writer.py defines AnalysisDBWriter, which ties the whole thing together: analyse a track, write its ANLZ files, and update master.db — all in one pipeline.

sequenceDiagram participant API as analyze-full route participant DW as AnalysisDBWriter participant AE as AnalysisEngine participant AW as ANLZ writer participant DB as master.db API->>DW: analyze_and_save(track_id, force) DW->>AE: run_full_analysis(path) AE-->>DW: bpm, key, beatgrid, waveforms, cues DW->>AW: write DAT / EXT / 2EX files DW->>DB: update bpm, key, analysed flag DW-->>API: result

It also offers analyze_batch() — a generator that yields a progress dict per track — and get_unanalyzed_tracks(), which finds tracks with a BPM of zero or missing ANLZ files. The batch path is what the /api/library/analyze-batch route streams to the UI as NDJSON.

11.9 Quality Presets and Caching

The user can trade speed against accuracy through three analysis quality presets:

PresetAnalyzersUse when
Fastlibrosa onlyA large batch where rough numbers are fine.
Standardmadmom beat trackingThe everyday default — accurate and reasonably quick.
ThoroughThe full ensembleA small set of tracks that must be exactly right.

And, as Chapter 10 described, every result is cached. The analysis cache is keyed on the file's modification time, size, and the analyzer version — so re-analysing an unchanged library is nearly free, and changing the analyzer invalidates every stale entry automatically.

11.10 The Native Counterpart

The Python engine described here is the high-accuracy, offline reference path. The Rust layer (Chapter 19) carries its own, lighter analysis — a RustFFT waveform computation and a Krumhansl-Schmuckler key detector — so the native side can draw a waveform and estimate a key without a round-trip to the Python backend. Python remains the accuracy reference; Rust is the low-latency convenience path. The boundary rule of Chapter 4 holds: realtime waveform rendering is Rust, the authoritative offline analysis is Python.

✦
Chapter summary

The analysis engine extracts BPM (madmom, librosa fallback), key (essentia, K-S/Temperley fallback), a full beatgrid, loudness, phrase and auto-cue markers, and seven waveform variants — all in ProcessPoolExecutor workers so the API never blocks. The AnalysisDBWriter orchestrator chains analysis to ANLZ files to a database update. Three quality presets and an aggressive cache let the engine scale from a single track to a whole library.
