Build, Packaging & Deployment

Turning three language stacks into one double-clickable desktop installer — the PyInstaller sidecar bundle, the Tauri build, the Windows installers, and the CI workflows that gate every commit.

22.1 The Packaging Problem

The application is three stacks in three languages. The end user must not see any of that. They download one installer, run it, and get one icon. The build pipeline's job is to collapse Python, JavaScript, and Rust into that single artefact — and, critically, to do it without requiring the user to install Python.

22.2 Bundling the Python Sidecar

The single hardest packaging task is the Python backend. It cannot ship as loose .py files — that would require the user to have the right Python and every dependency. Instead, PyInstaller compiles the entire backend, the interpreter, and every library into one self-contained executable named rb-backend.

FileRole
backend_entry.pyThe entry point PyInstaller compiles — it boots the FastAPI app exactly as python -m app.main would.
backend.specThe PyInstaller specification — what to include, hidden imports, data files, the output name.

The result is a binary the Tauri shell can spawn as a sidecar (Chapter 4). From the user's perspective there is no Python — there is just an application that happens to have a fast analysis engine inside it.

▲
Why bundling the analysis stack is hard

The backend depends on librosa, madmom, essentia, numpy, scipy, and numba — heavy scientific libraries with native extensions and non-obvious data files. Getting PyInstaller to find all of them is exactly what backend.spec exists to encode. This is the cost of choosing Python for the DSP layer — paid once, in the spec file.

22.3 Building the Frontend

The frontend build is the simplest of the three. npm run build runs Vite, which bundles, minifies, and tree-shakes the React application into a static frontend/dist directory. In a production Tauri build, the WebView loads this static bundle — not the Vite dev server. This is why a React-only change still requires a rebuild before it shows up in a Tauri WebDriver test.

22.4 The Tauri Build

npm run tauri build is the final assembly step. It compiles the Rust code in release mode, embeds the static frontend bundle, embeds the rb-backend sidecar as a bundled binary, and produces the platform installer. On Windows that means a .msi and an NSIS .exe.

22.6 Producing a Release

A release is produced by scripts/local-release.ps1, which drives the Windows build and packaging. Two deliberate facts about the release shape:

  • No code-signing. The installers are not code-signed, so Windows SmartScreen shows a warning on first run. The changelog states this openly — it is an expected, documented behaviour for an open, no-subscription project, not a defect.
  • SHA256SUMS published. Every release publishes the SHA-256 checksums of its artefacts, so a user can verify the download's integrity even without a code signature.

A second workflow, release.yml, handles release builds. There is also a pre-commit hook configuration (.pre-commit-config.yaml) that a developer installs once per machine; it runs the formatters, the type checker, and a set of guards — including one that fails the commit if a .env file or a master.db is staged.

22.8 The Auto-Format Hook

Beyond CI, an editor-level hook keeps the codebase clean as it is written. A PostToolUse hook runs the right formatter after any file edit — ruff format for Python, cargo fmt for Rust, Prettier for the frontend — so formatting is never a manual chore and never a review comment.

22.9 Deployment Shape

"Deployment" for this project is unusual: there is no server to deploy to. The application is installed, not deployed. The full picture:

AspectThis project
TargetThe user's own Windows desktop.
ArtefactA .msi or NSIS .exe installer.
Runtime prerequisitesFFmpeg on the PATH. Nothing else — Python is bundled.
UpdatesRe-run a new installer.
Server componentNone. The "backend" is a local sidecar process.
◆
The Docker files are vestigial

The repository still contains docker-compose.yml and two Dockerfiles. The Slopcode-Cleanup audit flagged them as dead — the project is a desktop application, not a containerised service — and marked them for removal. They are a historical artefact, not a deployment path.

✦
Chapter summary

The build pipeline collapses three language stacks into one installer: PyInstaller bundles the Python backend into a self-contained rb-backend executable, Vite builds the frontend to a static bundle, and tauri build assembles both with the release Rust binary into a Windows .msi and NSIS .exe. The user installs one artefact and needs only FFmpeg. A four-job CI workflow gates all three stacks plus a doc-drift check on every push, and an auto-format hook keeps the tree clean as it is written.
