Component Reference

A guided tour of the React component library — the feature views, the shared building blocks, the settings tabs, and the USB panels. The DAW and the waveform system have chapters of their own.

Legacy components, kept on purpose

Two components — LibraryView and ToolsView — still ship but are no longer mounted as tabs. MetadataView superseded the first as the library route; UtilitiesView superseded the second. They remain importable for reuse rather than being deleted outright.

15.3 The Shared UI Components

Beneath the views sits a layer of shared, non-lazy components used throughout the application.

ComponentPurpose
TrackTableThe reusable, sortable track table — Camelot-coloured keys, BPM display, multi-select, configurable columns, drag-reorder. Used by four different views.
PlaylistBrowserThe Rekordbox playlist tree — expand/collapse, drag-reorder, smart-playlist editing, context menu.
PlayerThe compact mini player at the bottom of the screen — play/pause, volume, seek; maximisable into the DAW.
ToastContextThe toast-notification provider — exposes useToast(). The replacement for alert().
ConfirmModalA promise-based replacement for window.confirm() — call it from anywhere, await a boolean.
PromptModalA promise-based replacement for window.prompt().
RenameModalAn inline rename dialog.
BatchEditBarThe batch-editing toolbar — rating, colour label, mass save — shown when tracks are selected.
ImportProgressBannerThe sticky bottom-of-screen banner aggregating import and download progress.
SmartPlaylistEditorThe rule-based smart-playlist editor modal, with a live preview.
MatchInspectorModalThe SoundCloud sync match inspector — per-track score and status.
SoundCloudProgressModalThe OAuth and download progress overlay.
shared/WaveformMiniCanvasA reusable canvas waveform renderer — CDJ-style 3-band colours, used by the DAW overview.

The three modal components — ConfirmModal, PromptModal, RenameModal — are worth noting as a pattern. Each is rendered once at the application root and exposes an imperative, promise-returning function. This is how the cleanup era removed every blocking native dialog while keeping calling code simple: await confirmModal({ … }) reads almost like the confirm() it replaced.

This container-plus-tab-bodies shape is a direct product of the Slopcode-Cleanup, which split a 1,179-line monolithic SettingsView into one focused component per tab. The same split produced the USB panels below.

15.6 The Component Contract

Every new component is held to a mandatory contract from the frontend rulebook. A component that fetches data must have explicit loading and error states:

const [loading, setLoading] = useState(false);
const [error, setError]     = useState(null);

if (loading) return <Spinner />;
if (error)   return <ErrorMessage message={error} />;

And a new view must be registered in the lazy-load tab router in main.jsx. Logging uses the shared dev-only logger (utils/log.js), not bare console calls. These rules are not suggestions — they are what keeps a 100-component application coherent.

15.7 A Note on the Cleanup's Legacy

It is worth appreciating how much of this chapter's structure exists because of the Slopcode-Cleanup. Before Phase 4 of that mission, UsbView was 1,937 lines and SettingsView was 1,179. Both are now thin containers delegating to focused sub-components — the settings/ and usb/ directories are the visible result. The same era split the waveform editor and the DAW, which Chapters 16 and 17 take up next.

Chapter summary

The component library is four groups: sixteen lazy-loaded feature views, a dozen shared building blocks (the reusable TrackTable, the player, the promise-based modals that replaced native dialogs), nine settings tabs, and seven USB panels. The settings and USB families are thin-container-plus-sub-components splits from the Slopcode-Cleanup. Every component honours a mandatory loading/error contract.
