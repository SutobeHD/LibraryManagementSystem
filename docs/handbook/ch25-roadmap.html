25.3 Theme 1 — Infrastructure Hardening

Two foundational features unblock the rest of the roadmap.

The database write-lock retrofit

An empirical audit (Chapter 10 mentioned it) found that the _db_write_lock auto-wrap is a hand-maintained method-name list that can silently drift — one mutator is already missing — and, more seriously, that the MyTags routes write through LiveRekordboxDB directly, bypassing the wrapped facade entirely. That is three unprotected concurrent write paths into a database that crashes on concurrent writes. The recommended fix replaces the drifting name list with a prefix-matched class decorator applied to both database classes, plus a CI test that fails if any mutator is left unwrapped.

Paired-device tokens — security Phase 2

Phase 1 (Chapter 21) ships a single boot-time session token. That cannot be handed to a phone, scoped per device, or revoked. Phase 2 introduces per-device 256-bit tokens with a QR-pairing flow, stored as SHA-256 hashes in a new sidecar-local auth.db that never touches master.db. The design adds just three routes — pairing start, pairing complete, and revoke — and keeps require_session's signature unchanged so all 84 existing route decorations stay byte-identical.

▲
A hard blocker

Paired-device tokens are a hard prerequisite for the mobile companion — with zero lines of code written today, the entire mobile feature is blocked behind this one. It is the single highest-leverage item on the roadmap.

25.4 Theme 2 — External Track Matching

Four features all need the same primitives — a fuzzy matcher, a version-tag parser, a fingerprint wrapper, an adapter registry. Rather than fork them three times, the cluster begins with one shared module.

The unified track-match module

A new flat module, app/external_track_match.py, lifts the fuzzy matcher out of the SoundCloud sync engine into reusable pure functions, adds a canonical twelve-value version-tag vocabulary (original, extended, radio, club, remix, bootleg, edit, mashup and more), and defines a source-adapter protocol. The SoundCloud engine becomes its first adapter; later milestones add a Discogs adapter and a bundled Chromaprint binary.

The remix detector

DJ libraries accumulate many variants of one work — a Radio Edit, an Extended Mix, a VIP, a bootleg — as unrelated rows. The remix detector classifies each track's variant type and links related variants, with a target of ≥95% classification precision. It builds in three milestones: a title-only classifier first (no network, no binary dependencies), then local clustering using the Rust fingerprint engine of Chapter 19, then an opt-in Chromaprint + AcoustID + MusicBrainz path.

The extended-remix finder

A DJ often owns the Radio Edit (2–3 minutes) when they need the Extended Mix (5–8 minutes) with the long intro for clean blending. This feature scans the library and surfaces, per track, an Extended version available on SoundCloud or elsewhere that the user does not yet own — with a Discogs ground-truth gate to avoid fruitless searches and a four-tier spam filter for the noisier sources.

The quality-upgrade finder

Libraries mix true lossless with MP3s — and some "FLAC" files are MP3-transcodes betrayed by a spectral cliff below the Nyquist frequency. This feature scores every track's quality (container, true bitrate, sample rate, bit depth, spectral cutoff) and detects transcodes with a target ≥95% precision. Its defining concern is blast radius — replacing a file must migrate every cue, beatgrid, tag, rating, and play count intact, so it ships in cautious phases beginning with a read-only probe.

25.5 Theme 3 — The Recommender Trilogy

The recommender is deliberately split into three documents, each a distinct engine, building on the last.

The rules baseline

A deterministic "what mixes well after this?" recommender — a harmonic-mixing assistant. It scores candidates on a continuous weighted blend of key, BPM, genre, MyTag overlap, and energy, using only data the library already has. BPM uses a Gaussian tuned to the CDJ-3000's ±6% pitch range; key uses a Camelot-wheel distance table. It also serves as the cold-start fallback and the baseline the taste-aware engine must beat.

The similar-tracks recommender

A purely offline "I like this — what else of mine sounds like it?" engine. It builds a roughly 46-dimensional handcrafted feature vector per track — BPM, key, loudness, four spectral descriptors, MFCC statistics, a chroma mean — stored in a sidecar database, and ranks by cosine similarity. A brute-force numpy comparison handles a 50,000-track library in tens of milliseconds; an approximate-nearest-neighbour index is reserved for libraries beyond 200,000.

The taste-aware engine

The most ambitious of the three: a recommender that learns a user's taste from observed behaviour — plays, skips, likes, tag patterns — and compares candidates with a learned audio embedding that captures timbre and production style beyond BPM and key. It starts with a centroid-based taste vector and benchmarks heavier music-embedding models (CLAP, MERT, OpenL3) against strict gates on extraction time and installer size, with an optional LLM explanation layer. It depends on its two sibling recommenders shipping first.

25.6 Theme 4 — Metadata Repair

Real-world libraries carry malformed metadata — the artist buried in the title's parentheses, feat. in the wrong field, reversed title/artist pairs, track-number prefixes, HTML entities, mojibake. The name fixer catalogues fourteen malformation classes, detects them with ≥90% recall, and applies deterministic high-precision transforms with a dry-run preview and a full audit log enabling revert. It is a maximal-blast-radius feature — it writes user metadata — so it ships detector-first, then the apply path, then optional MusicBrainz enrichment.

25.7 Theme 5 — The Mobile Companion

A thin mobile client, focused on Ranking mode — rating and tagging tracks from the couch or on the road. It is explicitly not standalone: all library state, audio, and database writes stay on the desktop app, and the phone is a remote UI that reaches it over the LAN or a Tailscale tunnel.

The chosen shape is a Progressive Web App on the existing Vite/React stack, targeting at least 60% component reuse of the existing Ranking view and a sub-200-KB first-paint bundle. It is gated entirely behind the paired-device token work of Theme 1 — which is why that infrastructure item leads the roadmap.

25.9 Beyond the Pipeline

A few items sit outside the formal pipeline as acknowledged post-MVP work in the changelog: manual hot-cue editing (auto-cues work today) and creating a master.db from scratch with no Rekordbox install present. The Design Lab view (Chapter 15) also gestures at longer-horizon ideas — stem separation, a set planner, a streaming hub — as UI mock-ups, deliberately not yet committed to the pipeline.

Chapter summary

Twelve researched features are in flight across five themes: two infrastructure items (a write-lock retrofit and the paired-device tokens that gate everything mobile), a four-feature track-matching cluster built on one shared module, a three-stage recommender trilogy, a metadata name fixer, and a PWA mobile companion. Each is a live pipeline document with measured benchmarks and milestone plans — and the authoritative status is always the pipeline dashboard, not this snapshot.
